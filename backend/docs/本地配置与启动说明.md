# 本地配置与启动说明

这份文档专门说明 Allo（元枢）当前本地开发环境里最常用、也最容易混淆的几件事：

- 模型怎么配置
- 搜索 / 抓取 / 生图怎么配置
- 怎么启动本地服务
- 本地如何跳过登录态

## 一、当前配置分成哪几层

你现在本地主要有三层配置：

### 1. `config.yaml`

这是主配置文件，负责定义：

- 模型列表
- 工具列表
- sandbox
- summarization
- memory

也就是说，Allo 知道“有哪些模型”和“有哪些工具”，主要看这里。

### 2. `.env`

这是项目根目录下的环境变量文件，主要放：

- API Key
- 第三方服务地址
- 本地数据库连接

### 3. `backend/.env`

这是后端侧额外补的一层 env 文件。

在你当前这套本地运行方式里，之所以还保留 `backend/.env`，主要是为了兼容一些直接从后端 / skill 脚本启动的场景。

## 二、模型怎么配置

模型配置在 `config.yaml` 里的 `models:` 段。

例如当前本地配置大致是：

```yaml
models:
  - name: gpt-5.4
    display_name: GPT-5.4
    use: langchain_openai:ChatOpenAI
    model: gpt-5.4
    api_key: $OPENAI_API_KEY
    base_url: https://kkcode.vip/v1
    max_tokens: 8192
    supports_vision: true
```

### 关键字段说明

- `name`
  - Allo 内部使用的模型标识
- `display_name`
  - 前端展示名称
- `use`
  - 使用哪个 LangChain provider
- `model`
  - 上游模型名
- `api_key`
  - API key，一般写成环境变量引用，例如 `$OPENAI_API_KEY`
- `base_url`
  - 如果你走 OpenAI 兼容网关，这个必须配置

### 当前推荐做法

如果你使用的是 OpenAI 兼容接口：

- 仍然用 `langchain_openai:ChatOpenAI`
- 然后在 `config.yaml` 里配置好：
  - `api_key`
  - `base_url`

## 三、搜索和抓取怎么配置

工具配置在 `config.yaml` 里的 `tools:` 段。

### 1. web_search

当前配置走的是 Tavily：

```yaml
tools:
  - name: web_search
    group: web
    use: deerflow.community.tavily.tools:web_search_tool
    max_results: 5
```

需要的 key：

- `TAVILY_API_KEY`

建议放在：

- `.env`
- 如有需要也同步放在 `backend/.env`

### 2. web_fetch

当前配置走的是 Jina：

```yaml
tools:
  - name: web_fetch
    group: web
    use: deerflow.community.jina_ai.tools:web_fetch_tool
    timeout: 10
```

需要的 key：

- `JINA_API_KEY`

同样建议放在：

- `.env`
- `backend/.env`

## 四、生图怎么配置

当前生图不是直接在 `config.yaml` 里配 provider，而是由 skill 脚本负责。

核心脚本是：

- `skills/public/image-generation/scripts/generate.py`

这个脚本现在已经改成按下面顺序尝试：

1. Gemini 代理
2. 官方 Gemini
3. Seedream 回退

### 1. Gemini 代理

如果你要优先走代理，需要配置：

```env
GEMINI_IMAGE_BASE_URL=https://api.gptoai.top
GEMINI_IMAGE_API_KEY=你的代理 key
GEMINI_IMAGE_MODEL=gemini-2.5-flash-image
```

### 2. 官方 Gemini

如果没有代理或代理不可用，会尝试官方 Gemini：

```env
GEMINI_API_KEY=你的官方 Gemini key
```

### 3. Seedream 回退

如果 Gemini 生图不可用，会自动回退到 Seedream：

```env
ARK_API_KEY=你的 ARK key
SEEDREAM_MODEL=doubao-seedream-5-0-260128
SEEDREAM_SIZE=2048x2048
```

### 当前这块的重要变化

`image-generation` 脚本现在会自己读取：

- 根目录 `.env`
- `backend/.env`

所以它不再完全依赖父进程有没有正确 `export` 环境变量。

这一步是专门为了解决之前文生图子进程拿不到 key、结果卡死在错误 provider 上的问题。

## 五、skill 怎么配置

当前内置 skills 目录是：

- `skills/public/`

运行时映射路径配置在 `config.yaml`：

```yaml
skills:
  container_path: /mnt/skills
```

你新增的 `tdli_style_slides` 现在已经迁移为内置 skill：

- `skills/public/tdli-style-slides/`

## 六、怎么启动 Allo

### 前台启动

```bash
make dev
```

会启动：

- frontend
- gateway
- LangGraph
- nginx

### 后台启动

```bash
make dev-daemon
```

日志会写到 `logs/` 目录。

### 停止服务

```bash
make stop
```

## 七、本地怎么跳过登录态

当前本地开发模式下，gateway 已经配置成 dev bypass。

也就是说，通过本地启动脚本启动时，会给 gateway 带上：

```env
ENV=development
SKIP_AUTH=1
```

这意味着：

- `/api/models`
- `/api/marketplace/*`
- 以及其他依赖 auth context 的 gateway 接口

在本地开发时不会强制你先走完整登录流程。

### 这块逻辑在哪里

- `backend/app/gateway/auth.py`

### 当前 dev 身份

当 `SKIP_AUTH=1` 生效时，默认会使用：

- `user_id=dev-user-000`
- `org_id=dev-org-000`
- `role=admin`

### 注意

`SKIP_AUTH=1` 只允许在：

- `development`
- `dev`
- `test`

环境下生效。

如果不是开发环境，代码会拒绝跳过登录。

## 八、本地 session fallback 是什么

当前还有一个本地 dev session 文件：

- `data/local-dev-auth-sessions.json`

这主要是给某些本地登录态恢复场景准备的。

但你现在本地联调最主要依赖的仍然是：

- `SKIP_AUTH=1`

## 九、当前比较重要的已知行为

### 1. models 接口之前报错的原因

前端之前有一个问题：

- `/api/models` 返回异常时，React Query 拿到 `undefined`
- 导致页面直接报错

现在已经修过：

- `frontend/src/core/models/api.ts`
- `frontend/src/core/config/api.ts`

### 2. 聊天页 hydration mismatch

之前工作区聊天页里：

- `react-resizable-panels`
- Radix 一些 trigger/button

在 SSR 和客户端 hydrate 时生成的 id 不一致，导致 hydration mismatch。

现在做了最小修复：

- `frontend/src/components/workspace/chats/chat-box.tsx`

挂载前不渲染那层 resizable shell，避免首屏 hydration 对比失败。

### 3. 文生图卡死问题

之前最大的问题是：

- 生图脚本作为子进程运行时，经常拿不到正确 env
- 然后错误地去打官方 Gemini
- 又因为连接卡住，单 worker 的 LangGraph 会被整个拖死

现在已经做了几件事：

- 脚本自己读取 `.env`
- 生图走 fallback chain
- 请求加 connect/read timeout

所以这条链比之前稳定很多。

## 十、推荐的本地使用方式

推荐你现在本地开发按这个方式来：

1. 模型和工具结构放在 `config.yaml`
2. 所有 key 放在 `.env`
3. 需要兼容后端脚本时，再镜像到 `backend/.env`
4. 启动前执行：

```bash
make stop
make dev
```

5. 出问题优先看日志：

- `logs/langgraph.log`
- `logs/gateway.log`
- `logs/frontend.log`
- `logs/nginx.log`

## 十二、排障建议

### models 接口 401

先确认是不是用下面方式启动的：

- `make dev`
- `make dev-daemon`

因为本地 auth bypass 是在 dev 启动链里处理的。

### web_search 鉴权失败

先看：

- `TAVILY_API_KEY`

### web_fetch 鉴权失败

先看：

- `JINA_API_KEY`

### 生图 / PPT 卡住

优先看：

- `GEMINI_IMAGE_BASE_URL`
- `GEMINI_IMAGE_API_KEY`
- `ARK_API_KEY`

再看：

- `skills/public/image-generation/scripts/generate.py`
- `logs/langgraph.log`

### 新对话也不回复

这通常表示：

- 某个长任务占住了单 worker 的 LangGraph

最快恢复方式：

```bash
make stop
make dev
```
