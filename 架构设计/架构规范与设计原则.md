# 架构规范与设计原则

更新时间：2026-04-01

## 一、目的

本文档用于统一 Allo / DeerFlow 当前多租户体系下的架构边界、设计约束与后续实现原则，避免在持续开发过程中再次出现：

- `gateway` 与 `harness` 边界混乱
- 租户上下文透传不一致
- 记录层、配置层、运行时层三套状态相互脱节
- 临时调试改动逐渐固化成长期架构负担

本文档的目标不是描述所有代码细节，而是定义后续开发必须遵守的架构规范。


## 二、总体架构原则

### 1. 严格分层

系统分为至少三层：

1. `Gateway / App` 层
   - 负责认证、数据库访问、业务记录、组织/用户隔离、管理 API
   - 模块命名空间：`app.*`

2. `Harness / Agent` 层
   - 负责 prompt 组装、tools/skills/MCP 运行时加载、agent 执行
   - 模块命名空间：`deerflow.*`

3. `Frontend` 层
   - 负责用户交互、管理视图、配置入口、调用 Gateway / LangGraph API

### 2. 依赖方向必须单向

依赖方向必须保持：

`frontend -> gateway -> harness abstraction`

而不是：

`harness -> gateway implementation`

具体要求：

- `deerflow.*` 不应直接长期依赖 `app.*`
- `harness` 只能依赖抽象接口、runtime config、或显式注入的数据
- `app.*` 可以实现并注册这些抽象接口

### 3. 运行时上下文比进程内耦合更重要

多租户系统中，真正驱动 runtime 行为的应该是：

- `user_id`
- `org_id`
- 当前请求的 runtime 决策数据

而不是让 `harness` 在内部主动回头读取 `gateway` 业务实现。


## 三、多租户设计原则

### 1. 用户上下文必须显式存在

任何会影响运行时结果的逻辑，只能基于显式用户上下文做判断。

允许的上下文来源：

- `configurable.user_id`
- `configurable.org_id`
- 或兼容别名 `x-user-id` / `x-org-id`

不允许：

- 使用全局单例状态猜测当前用户
- 使用某个共享缓存隐式推断当前租户

### 2. 读写都必须租户约束

一个资源是否真正租户化，不仅取决于“写入按用户/组织分表或分字段”，也取决于：

- 读取是否按 `user_id / org_id` 过滤
- 运行时是否按 `user_id / org_id` 决定最终可用集合

### 3. 隔离优先级

后续所有能力默认按以下优先级实现隔离：

1. 身份隔离（auth/session）
2. 数据隔离（threads, KB, soul, memory, api_keys）
3. 配置隔离（skills, MCP, model settings）
4. 运行时能力隔离（最终可用 skill/tool/MCP 集合）


## 四、配置类资源的统一模型

### 1. 区分“拥有资格”和“是否启用”

配置类资源不能把所有状态混成一张表。

必须区分：

- 是否属于当前用户/组织的候选集合
- 当前是否启用

例如：

- `install` = 加入候选集合
- `enable` = 候选集合中的激活状态

### 2. 最终运行时只认“决议结果”

不管底层状态存成几张表，最终 runtime 只能依赖统一决议结果。

也就是说：

- settings 页面
- prompt 组装
- runtime skill/tool 过滤

应该共享同一套 resolver，而不是各自拼自己的逻辑。

### 3. 不允许“记录层与运行时脱节”

如果系统存在：

- install 记录已落库
- 前端显示已安装
- 但 runtime 完全不受影响

那就说明架构没有收口，不能算完成。


## 五、Skill 设计原则

### 1. Skill 需要统一 catalog 视图

最终必须存在一个“用户 skill catalog 决议层”，输入至少包括：

- built-in 默认 skill
- marketplace 已安装 skill
- 用户/组织级 enable/disable toggle

输出：

- 当前用户最终拥有的 skill 列表
- 当前用户最终启用的 skill 列表

### 2. Skill install 与 enable 不是同一状态

不允许简单地把 install 直接等价成 enabled。

正确关系：

- install 决定是否进入候选集合
- enable 决定候选集合中哪些进入 runtime

### 3. 未安装 skill 不应出现在最终用户 catalog

如果 marketplace skill 没安装，按产品语义它就不该拥有候选资格。

所以：

- 未安装 skill 不应进入最终用户 skill 表单
- 更不应进入 prompt/runtime 上下文


## 六、Tool 设计原则

### 1. Tool 也必须有 runtime identity

marketplace tool 与 runtime tool 必须建立明确映射，例如：

- catalog item id
- display name
- runtime tool name

没有这个映射，就无法把 install 状态正确作用到 runtime。

### 2. Tool 的最终可用集合也必须可决议

工具最终是否进入 agent，可由这些因素共同决定：

- 平台内建能力
- marketplace install 状态
- 用户配置/启用状态
- 模型能力约束

但这些因素最终必须收敛到一个统一决议器，而不是散落在不同函数里各自判断。


## 七、MCP 设计原则

### 1. MCP 的核心不是“安装”，而是“配置 + 上下文隔离”

MCP 与 skill/tool 的不同点在于：

- 它更偏连接器 / 外部能力入口
- 核心问题不是 catalog install，而是：
  - 当前用户有哪些 MCP 配置
  - 用什么凭据
  - runtime 实际用哪个上下文执行

### 2. 不要求物理实例一人一份，但要求逻辑上下文一人一份

多个用户即使使用同一个 MCP 模板，也必须保证：

- 配置隔离
- 凭据隔离
- cache / tool session 隔离

### 3. MCP runtime 应以当前用户配置为准

runtime 加载 MCP 时，应只依赖：

- 当前用户/组织配置
- 当前 runtime 上下文

而不是全局文件配置。


## 八、Gateway 与 Harness 的边界规范

### 1. Gateway 负责业务事实

Gateway 负责定义并持久化：

- auth/session
- user/org
- threads/runs
- install 记录
- toggle 记录
- knowledge bases
- soul / memory / api keys

### 2. Harness 负责执行，不负责自行决定业务事实来源

Harness 应该消费：

- runtime 上下文
- 抽象 store
- 或显式传入的已决议 runtime 数据

不应该：

- 在 `deerflow.*` 内直接长期 import `app.*` 业务实现
- 在 harness 初始化阶段偷偷做 gateway 进程级注册逻辑

### 3. 调试桥接可以存在，但不能成为长期设计

为了调通链路，可以临时：

- 在前端把 `user_id / org_id` 放进 configurable
- 在代码里加 runtime debug log

但最终应评估并回收这些桥接手段，避免它们变成隐式架构依赖。


## 九、前端设计原则

### 1. 前端显示语义必须与后端状态模型一致

前端不应把：

- install
- enable
- configure

混成一种操作。

建议语义：

- built-in skills: `Enable / Disable`
- marketplace items: `Install / Uninstall`
- MCP: `Configure / Enable / Disable`

### 2. 错误必须显示真实原因

前端错误处理不能把真实后端错误统一吞成模糊文案。

至少应透出：

- `detail`
- `message`
- 关键状态码语义


## 十、验证原则

### 1. 记录层通过不等于功能层通过

以下两者必须分别验证：

- 记录是否成功落库
- runtime 是否真的因此改变行为

### 2. 任何“租户化完成”都必须有 A/B 对照验证

至少验证：

- A 能看到自己的数据
- B 看不到 A 的数据
- B 直接访问 A 的资源返回 404/403

### 3. 配置类资源必须做三段验证

对于 install/enable 类资源，后续必须验证：

1. 安装前不可用
2. 安装后可用
3. 卸载后不可用


## 十一、当前工程上的重点提醒

### 1. 不要让管理页状态与 runtime 状态分裂

如果：

- settings 看起来是 A
- marketplace 看起来是 B
- runtime 实际是 C

那说明设计还没收口。

### 2. 不要把“全局默认值”误当成“最终用户视图”

全局模板、默认配置、系统 seed 数据，只能作为输入，不应直接等同于最终用户可用集合。

### 3. 任何新能力都应先问一个问题

这个能力的最终可用集合，是在哪个 resolver 里被统一决议的？

如果回答不出来，说明架构还不完整。


## 十二、最终目标

后续理想架构应达到：

- 一个用户一个最终能力视图
- 一个组织一个安装视图
- 一个 runtime 一个明确的上下文决议结果
- 所有管理页与 runtime 共用同一套状态模型

最终，系统不应再区分“看起来装了”和“实际能不能用”这两种状态。
