# Allo 全链路数据流澄清

本文档按真实时间线，从服务启动到用户完成一次完整会话，逐阶段描述数据如何在前端、Gateway、LangGraph、PostgreSQL、Redis、Sandbox 之间流转。

---

## 阶段 0：服务启动

服务器上有 6 个常驻进程 + 按需启动的 Docker 容器：

```
systemd 管理：
  ├── nginx          (80端口，反代所有服务)
  ├── postgresql     (5432，数据持久化)
  ├── redis          (6379，session缓存/限流)
  ├── allo-gateway   (8001，FastAPI 业务网关)
  ├── allo-frontend  (3000，Next.js 前端)
  └── allo-langgraph (2024，LangGraph agent 引擎)

Docker 按需：
  └── allo-sandbox-* (用户代码执行容器，空闲销毁)
```

启动顺序：`postgresql → redis → allo-gateway → allo-langgraph → allo-frontend → nginx`

### Gateway 启动时

1. 连接 PostgreSQL，执行 `alembic upgrade head` 确保 schema 最新
2. 连接 Redis，初始化连接池
3. 加载 `config.yaml` 平台配置（模型、搜索引擎、默认 skills）
4. 注册所有 router（auth, users, threads, memory, mcp, skills, agents, soul, api_keys, uploads, artifacts, suggestions）
5. 启动 sandbox manager 后台任务（定期扫描空闲容器）

### LangGraph 启动时

1. 读取 `langgraph.json`，注册 `lead_agent` graph
2. 调用 `make_checkpointer()` 连接 PostgreSQL（checkpoint 存储）
3. 等待请求（ASGI 异步服务）
4. 不主动加载任何用户数据，一切都是 per-request 动态加载

### nginx 反代规则

```
/              → frontend:3000
/api/langgraph → langgraph:2024（带 auth_request 鉴权）
/api/*         → gateway:8001
```

## 阶段 1：用户打开浏览器

```
用户浏览器 → http://服务器IP
  → nginx 匹配 / → proxy_pass http://127.0.0.1:3000
  → Next.js 返回 HTML/JS/CSS
  → 浏览器渲染页面
```

### Next.js middleware 拦截（服务端执行）

```
每个请求进来：
  ├── 路径是 /workspace/* 或 /admin/* ？
  │   ├── 是 → 检查 cookie 中有没有 session_token
  │   │   ├── 有 cookie → 放行，继续渲染页面
  │   │   └── 没有 cookie → 302 重定向到 /login
  │   └── 否（/login、/register、/、静态资源等）→ 直接放行
```

注意：middleware 只检查 cookie 是否存在，不验证 cookie 是否有效。这是故意的：
- middleware 跑在 Next.js 服务端，如果每次都调 Gateway 验证 session，会增加延迟
- 真正的 session 验证在后续 API 请求时由 Gateway 完成
- 如果 cookie 过期了，用户能进入 workspace 页面，但第一个 API 请求就会返回 401，前端再跳转到 login

## 阶段 2：认证流程

### 场景 A：新用户注册

```
浏览器 /register 页面
  → 用户填写邮箱 + 密码 + 显示名
  → 前端调 POST /api/auth/register { email, password, display_name }
  → nginx → Gateway:8001

Gateway 内部处理：
  1. 校验邮箱格式、密码强度、邮箱是否已注册
  2. bcrypt 哈希密码 → password_hash
  3. 生成 user_id (UUID v4)
  4. 事务内执行：
     INSERT INTO users (id, email, password_hash, display_name)
     INSERT INTO organizations (id, name, slug='user-{user_id}')
     INSERT INTO organization_members (user_id, org_id, role='admin')
  5. 生成 session token: secrets.token_urlsafe(32)
  6. INSERT INTO sessions (id, user_id, token, expires_at=now+7天)
  7. 写 Redis: SETEX session:{token} 300 {AuthContext JSON}
  8. Set-Cookie: session_token={token}; HttpOnly; Path=/; SameSite=Lax; Max-Age=604800

→ 返回 200 { user_id, email, display_name }
→ 前端 router.push('/workspace/chats/new')
```

为什么注册时就创建 Organization：
- 隔离模型是 `user_id + org_id`
- 每个用户至少属于一个 org（自己的个人空间）
- 后续做团队功能时，用户可以被邀请加入其他 org
- 第一版每个用户只有一个个人 org

### 场景 B：老用户登录

```
浏览器 /login 页面
  → 前端调 POST /api/auth/login { email, password }
  → Gateway:
      1. SELECT * FROM users WHERE email = ? AND is_active = true
      2. bcrypt.verify(password, user.password_hash)
      3. 通过 → 创建 session（同注册步骤 5-8）
      4. 失败 → 401 "邮箱或密码错误"
  → Set-Cookie → 前端跳转 workspace
```

### 场景 C：后续每个 API 请求的 session 验证

这是最频繁的路径，每个需要认证的 API 都会走：

```
前端发任何 API 请求（比如 GET /api/threads）
  → 浏览器自动带上 Cookie: session_token=xxx
  → nginx → Gateway:8001
  → Gateway auth middleware (Depends(get_auth_context)):

      1. 从 cookie 读 token
         → 没有 token → 401

      2. 查 Redis: GET session:{token}
         → 命中 → 直接反序列化为 AuthContext(user_id, org_id, role)
         → 跳到步骤 5

      3. Redis 未命中 → 查 PG:
         SELECT * FROM sessions WHERE token = ? AND expires_at > now()
         → 没找到或已过期 → 401

      4. 找到有效 session → 查用户和 org:
         SELECT * FROM users WHERE id = session.user_id
         SELECT * FROM organization_members WHERE user_id = ?
         → 构建 AuthContext(user_id, org_id, role)
         → 写回 Redis: SETEX session:{token} 300 {AuthContext JSON}

      5. 把 AuthContext 挂到 request.state.auth
         → 后续 handler 通过 Depends(get_auth_context) 拿到
         → 所有业务查询都带 WHERE user_id = auth.user_id
```

Redis 缓存的意义：
- 大部分请求在 Redis 就命中了（TTL 5 分钟内）
- 避免每个请求都查 PG
- 50-200 人并发时，Redis 能扛住，PG 压力小很多

### 场景 D：LangGraph 流式链路的鉴权（nginx auth_request）

这条链路比较特殊，因为流式请求不经过 Gateway，直接到 LangGraph。

所有用户共享同一个 LangGraph 进程，但每次请求通过 nginx 注入的 `X-User-Id` 区分用户身份。LangGraph 内部根据这个 user_id 动态加载该用户的 skills、memory、soul、MCP 等数据，构建该用户专属的 graph 实例。请求结束后实例销毁，不会残留给下一个用户。

```
前端 SDK → POST /api/langgraph/threads/{id}/runs/stream
  → 到达 nginx

nginx 配置：
  location /api/langgraph/ {
      auth_request /internal/auth-check;
      auth_request_set $x_user_id $upstream_http_x_user_id;
      auth_request_set $x_org_id  $upstream_http_x_org_id;

      proxy_pass http://127.0.0.1:2024;
      proxy_set_header X-User-Id $x_user_id;
      proxy_set_header X-Org-Id  $x_org_id;
  }

  location = /internal/auth-check {
      internal;
      proxy_pass http://127.0.0.1:8001/api/auth/check;
      proxy_pass_request_body off;
      proxy_set_header Cookie $http_cookie;
  }

Gateway /api/auth/check 端点：
  → 走和场景 C 一样的 session 验证
  → 验证通过 → 200 + 响应头 X-User-Id: xxx, X-Org-Id: xxx
  → 验证失败 → 401（nginx 直接拒绝原始请求）
```

## 阶段 3：Workspace 初始化

用户登录成功进入 `/workspace` 后，前端并行发起多个请求加载用户数据：

```
React 渲染 WorkspaceLayout
  │
  │  并行请求（都带 session cookie，都经过 Gateway auth）：
  │
  ├── GET /api/auth/session
  │   → Gateway 返回当前用户信息
  │   → { user_id, email, display_name, org_id }
  │   → 前端存入 React context，全局可用
  │
  ├── GET /api/threads?limit=50&sort=last_active_at
  │   → Gateway 查业务表：
  │     SELECT id, title, status, agent_name, last_model_name, last_active_at
  │     FROM threads WHERE user_id = ? AND status != 'deleted'
  │     ORDER BY last_active_at DESC LIMIT 50
  │   → 返回当前用户的 thread 列表
  │   → 前端渲染左侧 sidebar
  │   → 注意：不调 LangGraph，纯业务表查询
  │
  ├── GET /api/models
  │   → Gateway 合并两个来源：
  │     a. config.yaml 中的平台模型列表（所有用户可见）
  │     b. user_api_keys 表中该用户配置的 BYOK provider
  │   → 返回 [{ name, provider, is_platform, is_byok }]
  │   → 前端渲染模型选择器
  │
  ├── GET /api/agents
  │   → Gateway 合并两个来源：
  │     a. 平台默认 agents
  │     b. user_agents 表中该用户的自定义 agents
  │   → 返回 [{ name, description, model, is_platform, is_custom }]
  │   → 前端渲染 agent 选择器
  │
  └── localStorage 读取 LocalSettings
      → 上次选择的 model_name、mode、agent_name
      → 纯前端，不调后端
      → 用于恢复用户上次的偏好设置
```

前端拿到这些数据后渲染：
- 左侧 sidebar：thread 列表（按 last_active_at 排序）
- 顶部/底部：model 选择器、mode 选择器
- 主区域：如果有 thread_id 参数则打开对应 thread，否则显示 new chat 页面

## 阶段 4：新建对话

```
用户点击 sidebar 的"新建对话"按钮
  → 前端路由跳转到 /workspace/chats/new
  → useThreadChat() hook 执行：
      const threadId = uuid()        // 前端生成 UUID
      const isNewThread = true       // 标记为新 thread
  → useThreadStream({ threadId: undefined })
      // 注意：传 undefined，SDK 此时不创建 thread
  → 页面渲染：空白聊天区 + 输入框 + 模型/mode 选择器
```

此时：
- Gateway 没有任何操作
- LangGraph 没有任何操作
- PostgreSQL 没有任何写入
- 只是前端生成了一个 UUID 等着用

thread 真正被创建是在用户发第一条消息时（阶段 5）。

## 阶段 5：发送第一条消息

这是整个链路最密集的一步，5 个子步骤串行执行。

```
用户在输入框输入文字，点击发送
  │
  ├── Step 1: 乐观渲染（纯前端，0ms）
  │   → 立即在聊天区显示用户消息气泡
  │   → 不等任何后端响应
  │   → 用户体验上感觉"瞬间发出去了"
  │
  ├── Step 2: 上传文件（如有附件）
  │   → POST /api/threads/{threadId}/uploads (multipart/form-data)
  │   → Gateway 存到 users/{user_id}/threads/{threadId}/user-data/uploads/
  │   → 返回文件元数据 [{ name, path, size, mime_type }]
  │   → 没有附件则跳过
  │
  ├── Step 3: 创建 thread（仅第一条消息时）
  │   → POST /api/threads { thread_id, agent_name, default_model }
  │   → Gateway:
  │       校验 auth → 拿到 user_id
  │       调 LangGraph POST /threads → 创建 checkpoint 容器
  │       INSERT INTO threads 业务表
  │       创建文件目录:
  │         users/{user_id}/threads/{threadId}/user-data/workspace/
  │         users/{user_id}/threads/{threadId}/user-data/uploads/
  │         users/{user_id}/threads/{threadId}/user-data/outputs/
  │         users/{user_id}/threads/{threadId}/user-data/tmp/
  │       返回 { thread_id }
  │
  ├── Step 4: 创建 thread_run + 解析 Key
  │   → POST /api/threads/{threadId}/runs { model_name, agent_name }
  │   → Gateway:
  │       根据 user_id + model_name 解析 API Key:
  │         查 user_api_keys → 有则解密用户 Key
  │         没有 → 回退到 config.yaml 平台 Key
  │         都没有 → 返回错误
  │       INSERT INTO thread_runs (status=running)
  │       UPDATE threads SET last_active_at, last_model_name
  │       把解密后的 Key 写入 Redis: SETEX run:{run_id}:key 300 {key}
  │       返回 { run_id }
  │
  └── Step 5: 发消息 + 开始流式
      → SDK useStream.submit({
          messages: [{ type: "human", content: [...] }]
        }, {
          threadId,
          streamSubgraphs: true,
          config: { recursion_limit: 1000 },
          context: {
            thread_id, run_id, model_name,
            thinking_enabled, is_plan_mode, subagent_enabled, agent_name
          }
        })
      → SDK 内部调: POST /api/langgraph/threads/{threadId}/runs/stream
      → nginx auth_request 校验 → 注入 X-User-Id/X-Org-Id
      → 转发到 LangGraph:2024
      → 进入阶段 6
```

### Key 传递的安全设计

用户的 API Key 不经过前端。链路：

1. Gateway 在 Step 4 解析出 Key 后，写入 Redis: `SETEX run:{run_id}:key 300 {api_key}`
2. LangGraph 的 `make_lead_agent(config)` 从 `runtime.context.run_id` 拿到 run_id
3. 从 Redis 读取 Key: `GET run:{run_id}:key`
4. 用这个 Key 创建模型实例
5. run 结束后 Key 自动过期（TTL 5min）

Key 只在 Gateway → Redis → LangGraph 之间流转，不经过浏览器。

## 阶段 6：Agent 初始化与执行

LangGraph Server 收到 `POST /threads/{id}/runs/stream` 后，开始构建和执行 agent。

### 6.1 恢复 thread 状态

```
从 PostgreSQL checkpointer 读取该 thread_id 的最新 checkpoint
  → 第一条消息：checkpoint 为空，初始化空的 ThreadState:
    { messages: [], title: "", artifacts: [], todos: [], sandbox: null }
  → 后续消息：checkpoint 里已有之前的消息历史和状态
  → 把用户新消息追加到 messages 末尾
```

### 6.2 调用 make_lead_agent(config)

这个函数每次 run 都会被调用，构建一个全新的 graph 实例。即使同一用户、同一 thread、同一 model，也会重建。这不是浪费——graph 构建只要几十毫秒，真正耗时的是 LLM API 调用（秒级）。每次重建的好处是用户改了任何配置都能立即生效。

### user_id 传递链路

```
浏览器 cookie
  → nginx auth_request → Gateway 验证
  → nginx 注入 X-User-Id header
  → LangGraph Server 收到
  → 放入 config["configurable"]["x-user-id"]
  → make_lead_agent(config) 读取
  → 传给 load_skills / load_memory / load_soul 等函数
  → 同时放入 runtime.context，middleware 和工具也能读取
```

前端不需要显式传 user_id（也不应该，因为前端传的可以被伪造）。user_id 完全由服务端链路保证。

### 6.3 按 user_id 加载用户数据

```
make_lead_agent(config) 内部：
  │
  ├── 解析模型 Key
  │   → 从 Redis: GET run:{run_id}:key
  │   → 拿到 api_key + base_url
  │   → create_chat_model(name=model_name, api_key=..., base_url=...)
  │
  ├── 加载用户 Skills（per-user）
  │   → load_skills(user_id):
  │       扫描 skills/public/ (平台公共)
  │       + 扫描 users/{user_id}/skills/custom/ (用户私有)
  │       + 从 PG 读取该用户的 skill 开关状态
  │       → 返回合并后的 enabled skills 列表
  │
  ├── 加载用户 MCP 工具（per-user）
  │   → get_mcp_tools(user_id):
  │       平台默认 extensions_config.json
  │       + 用户 DB 覆盖 (user_mcp_configs 表)
  │       → 合并后加载 MCP server → 返回工具列表
  │
  ├── 加载用户 Soul（per-user）
  │   → 查 PG: user_souls WHERE user_id = ?
  │   → 没有则回退到平台默认 soul 模板
  │
  ├── 加载用户 Memory（per-user）
  │   → 查 PG: user_memory WHERE user_id = ?
  │   → 查 PG: user_memory_facts WHERE user_id = ? LIMIT 15
  │
  ├── 构建 system prompt
  │   → apply_prompt_template():
  │       基础角色设定
  │       + soul 内容
  │       + memory 上下文
  │       + skills 列表（name, description, location, root）
  │       + 工具使用说明
  │       + mode 指令（plan mode 等）
  │
  ├── 收集所有工具
  │   → 内置: web_search, crawl, read_file, write_file, bash, ls, ...
  │   → MCP: 从上面加载的
  │   → 特殊: task (子任务), present_file (artifact), ask_clarification
  │
  ├── 构建 middleware 链（15 个，按顺序）
  │   → ThreadDataMiddleware   — 创建 thread 文件目录
  │   → UploadsMiddleware      — 把上传文件路径注入 prompt
  │   → SandboxMiddleware      — 获取/释放 sandbox 容器
  │   → DanglingToolCallMiddleware
  │   → ToolErrorHandlingMiddleware
  │   → SummarizationMiddleware
  │   → TodoMiddleware (plan mode)
  │   → TitleMiddleware        — 首次交互后生成标题
  │   → MemoryMiddleware       — 会话结束后更新 memory
  │   → ViewImageMiddleware
  │   → DeferredToolFilterMiddleware
  │   → SubagentLimitMiddleware
  │   → ResearchBudgetMiddleware
  │   → LoopDetectionMiddleware
  │   → ClarificationMiddleware
  │
  └── 构建 StateGraph
      → create_agent(model, tools, middleware, system_prompt, state_schema=ThreadState)
      → 返回 CompiledStateGraph
```

### 6.4 harness 层需要的改动

harness 代码需要改动，但改动性质是"接收 user_id 参数"，不是"塞入业务逻辑"：

```
必须改的（接收 user_id 参数）：
  ├── load_skills(user_id)          — 现在是 load_skills()，无参数
  ├── load_memory(user_id)          — 现在读全局 memory.json
  ├── load_soul(user_id)            — 现在读全局 USER.md
  ├── get_mcp_tools(user_id)        — 现在读全局 extensions_config.json
  ├── _resolve_skills_path(user_id) — 现在映射到全局 skills 目录
  └── UserContext 数据类             — 新增，从 RunnableConfig 提取 user_id

不需要改的（保持原样）：
  ├── agent graph 构建逻辑（StateGraph、节点、边）
  ├── middleware 框架（before_agent/after_agent 机制）
  ├── 工具定义和注册
  ├── prompt 模板结构
  └── model 调用逻辑（create_chat_model）
```

跨层策略：harness 定义抽象接口（如 `MemoryStore`），app 层提供 PG 实现。harness 不 import app 层代码。无 user_id 时回退到现有行为（兼容本地开发）。

### 6.5 Graph 执行流程

```
START
  ↓
before_agent middlewares (ThreadData → Uploads → Sandbox)
  ↓
before_model middlewares (ViewImage → DeferredToolFilter)
  ↓
┌→ model 节点
│   → 把 system_prompt + messages 发给 LLM API
│   → LLM 返回文本 或 tool_calls
│     ↓
│   after_model middlewares (Title → SubagentLimit → ResearchBudget → LoopDetection)
│     ↓
│   LLM 返回了 tool_calls？
│     ├── 是 → tools 节点（执行工具）→ 结果追加到 messages → 回到 model ─┘
│     └── 否 → 跳出循环
│
after_agent middlewares (Memory)
  ↓
END
```

这个循环可能执行多轮。比如用户问"搜索最新的AI新闻并总结"：
- 第1轮: model 决定调 web_search → 执行搜索 → 结果回来
- 第2轮: model 看到搜索结果，决定调 crawl 抓取详情 → 执行 → 结果回来
- 第3轮: model 看到所有信息，生成最终回复 → 无 tool_calls → 结束

每一轮执行后：
- checkpointer 把当前 state 写入 PostgreSQL
- SSE 事件推送给前端（进入阶段 7）

## 阶段 7：流式响应回传

Graph 执行过程中，每个节点完成后产生 SSE 事件，实时推送给前端。

### SSE 事件类型

| 事件类型 | 内容 | 前端处理 |
|---------|------|---------|
| messages-tuple | 增量消息片段（token 级流式） | 逐 token 渲染 AI 回复 |
| values | 完整 state 快照 | 更新 thread.values（title, artifacts, todos） |
| updates | state 差量更新 | onUpdateEvent 回调（检测 title 变化） |
| events | LangChain 回调事件 | onLangChainEvent（tool 执行完成通知） |
| custom | 自定义事件 | onCustomEvent（子任务进度更新） |

### 传输链路

```
LangGraph:2024 → SSE stream → nginx (proxy_buffering off) → 浏览器

nginx 关键配置：
  proxy_buffering off;     ← 不缓冲，直接透传
  proxy_cache off;         ← 不缓存
  proxy_read_timeout 300s; ← 长超时（agent 可能跑很久）
```

### 前端 SDK 解析

```
useStream 内部自动处理：
  ├── 解析 messages-tuple → 更新 thread.messages → 逐 token 显示 AI 回复
  ├── 解析 values → 更新 thread.values
  ├── 解析 updates → onUpdateEvent 回调
  ├── 解析 events → onLangChainEvent 回调
  ├── 解析 custom → onCustomEvent 回调
  └── 更新 thread.isLoading / thread.status

React 渲染：
  → MessageList 组件 re-render
  → 每条 AI 消息逐 token 追加显示
  → 工具调用显示为折叠卡片（搜索结果、代码执行结果等）
  → artifacts 显示在右侧面板
  → todos 显示为任务列表（plan mode）
```

## 阶段 8：标题生成与同步

```
首次交互完成后（1条人类消息 + 1条AI回复）：
  │
  ├── TitleMiddleware.aafter_model() 自动触发
  │   → 用轻量 LLM 调用生成标题（比如"关于AI新闻的讨论"）
  │   → 写入 ThreadState.title
  │   → checkpointer 持久化到 PostgreSQL
  │   → SSE updates 事件: { title: "关于AI新闻的讨论" }
  │
  ├── 前端 onUpdateEvent 收到 title
  │   → sidebar 立即显示新标题（乐观更新 React Query 缓存）
  │   → 额外调 Gateway: PATCH /api/threads/{threadId} { title }
  │
  └── Gateway 更新业务表
      → UPDATE threads SET title = ? WHERE id = ? AND user_id = ?
```

为什么要双写（checkpoint + 业务表）：
- checkpoint 里的 title 是 LangGraph 运行态数据，恢复会话时用
- 业务表里的 title 是列表展示用，不需要每次都去 LangGraph 查 checkpoint

## 阶段 9：后续消息（同一 thread 继续对话）

与第一条消息的区别：

```
用户在同一个 thread 继续发消息
  │
  ├── 跳过 Step 3（thread 已存在）
  ├── 执行 Step 4（新建 thread_run + 解析 Key）
  │   → 因为用户可能切换了 model 或 mode
  │   → 每次发消息都是一次新的 run
  ├── 执行 Step 5（SDK submit）
  │
  └── LangGraph 处理：
      → 从 checkpointer 恢复最新 state（包含之前所有消息）
      → 新消息追加到 messages 末尾
      → make_lead_agent 重新构建 graph（加载最新 skills/memory/soul）
      → 执行 graph → 流式返回
      → TitleMiddleware 不再触发（title 已存在）
      → MemoryMiddleware 继续更新 memory
```

### model 切换场景

```
用户在 thread 中途切换 model（比如从 gpt-4o 切到 deepseek）
  → 前端 LocalSettings.context.model_name 更新
  → 下一次 sendMessage 时:
      Step 4 创建新 thread_run，model_name = 新模型
      Gateway 解析新模型对应的 Key
      LangGraph make_lead_agent 用新模型构建 graph
  → 对话历史不变（从 checkpoint 恢复），只是执行引擎换了
  → threads.last_model_name 更新为新模型
```

### 断线重连场景

```
用户刷新页面 / 网络断开后恢复
  → useStream({ reconnectOnMount: true })
  → SDK 检测到该 thread 有活跃的 run
  → 调 GET /api/langgraph/threads/{id}/runs/{runId}/stream 重新加入 SSE
  → 从断点继续接收流式数据
  → 如果 run 已结束，SDK 直接加载最终 state
```

## 阶段 10：代码执行（Sandbox）

```
Agent 决定执行代码（比如用户说"用 Python 画一个正弦波"）
  │
  ├── 1. SandboxMiddleware.before_agent 已在阶段 6 获取 sandbox
  │   → acquire(user_id) → 从用户池中拿一个容器
  │   → 如果用户已有空闲 sandbox → 复用
  │   → 如果没有且未达上限(2个) → 创建新容器:
  │       docker run -d \
  │         --name allo-sandbox-{user_id}-{short_id} \
  │         --memory 512m --cpus 0.5 \
  │         --network none --read-only --pids-limit 100 \
  │         -v {thread_workspace}:/mnt/user-data/workspace:rw \
  │         -v {thread_uploads}:/mnt/user-data/uploads:ro \
  │         -v {thread_outputs}:/mnt/user-data/outputs:rw \
  │         -v {thread_tmp}:/mnt/user-data/tmp:rw \
  │         -v {user_custom_skills}:/mnt/skills/custom:ro \
  │         -v {platform_public_skills}:/mnt/skills/public:ro \
  │         -e TMPDIR=/mnt/user-data/tmp \
  │         -w /mnt/user-data/workspace \
  │         allo-sandbox:latest
  │
  ├── 2. Agent 输出 tool_call: bash("python3 script.py")
  │   → bash tool 收到命令
  │   → 虚拟路径验证:
  │       /mnt/user-data/* → 允许（thread 私有区）
  │       /mnt/skills/*    → 允许读（公共+用户私有 skill）
  │       其他路径         → 拒绝
  │   → 路径翻译（虚拟 → host）:
  │       /mnt/user-data/workspace → users/{user_id}/threads/{thread_id}/user-data/workspace
  │       /mnt/skills/public/foo   → skills/public/foo
  │       /mnt/skills/custom/bar   → users/{user_id}/skills/custom/bar
  │   → 在 sandbox 容器内执行命令
  │   → 捕获 stdout/stderr
  │   → 输出中的 host 路径被掩码回虚拟路径
  │
  ├── 3. Agent 可能继续调用更多工具
  │   → write_file → 写到 /mnt/user-data/workspace/
  │   → read_file → 读 skill 脚本
  │   → bash → 再次执行
  │   → present_file → 注册 artifact
  │
  └── 4. SandboxMiddleware.after_agent 标记 sandbox 为 idle
      → 启动 idle_timeout 计时器（默认 10min）
      → 超时后容器被 sandbox_manager 后台任务销毁
      → 同一用户下次执行代码时，如果容器还在 → 直接复用
```

### Skill 脚本执行场景

```
Agent 读取 SKILL.md 后决定执行 skill 附带的脚本:
  → read_file /mnt/skills/public/ppt-generation/scripts/generate.py
  → bash "python3 /mnt/skills/public/ppt-generation/scripts/generate.py \
          --output /mnt/user-data/workspace/output.pptx"
  → 脚本从 /mnt/skills/public/ 读取（只读）
  → 输出写到 /mnt/user-data/workspace/（可写）
  → 用户私有 skill 同理，路径前缀为 /mnt/skills/custom/
```

## 阶段 11：Memory 更新

```
Agent 完成一次完整交互后（after_agent 阶段）：
  │
  ├── MemoryMiddleware.after_agent() 触发
  │   → 把本次对话内容推入 MemoryUpdateQueue
  │   → Queue 有 debounce 机制（不是每条消息都立即写，而是攒一批）
  │
  ├── MemoryUpdateQueue 触发写入
  │   → 调用 MemoryStore.save_memory(user_id, data)
  │   → PostgresMemoryStore:
  │       UPSERT INTO user_memory (user_id, context_json, updated_at)
  │       INSERT INTO user_memory_facts (user_id, content, category, confidence, source)
  │
  └── 下次该用户发消息时
      → make_lead_agent 构建 prompt 时
      → 从 PG 读取 user_memory WHERE user_id = ?
      → 注入到 system prompt 的 memory 段落
      → agent 就"记住"了之前的上下文
```

Memory 的隔离保证：
- 用户 A 的 memory 只写入 user_id=A 的行
- 用户 B 的 agent 构建 prompt 时只读 user_id=B 的 memory
- 不存在跨用户 memory 泄露的可能

## 阶段 12：会话结束与清理

```
一次完整的 run 结束后，各层的收尾动作：

前端：
  ├── onFinish 回调触发
  │   → PATCH /api/threads/{threadId}/runs/{runId} { status: 'succeeded' }
  │   → invalidate React Query 缓存（sidebar 刷新）
  │   → 请求 follow-up suggestions:
  │     POST /api/threads/{threadId}/suggestions
  │     → Gateway 用轻量 LLM 生成 3 条追问建议
  └── 用户可以继续发消息（回到阶段 9）或关闭页面

Gateway：
  ├── 更新 thread_runs: status=succeeded, finished_at=now
  ├── 更新 threads: last_active_at=now, last_model_name=实际使用的模型
  └── Redis 中 run:{run_id}:key 自动过期（TTL 5min）

LangGraph：
  ├── checkpointer 已在执行过程中持续写入 PostgreSQL
  ├── 最终 state 已持久化（包含完整消息历史、title、artifacts、todos）
  └── graph 实例被 GC 回收（每次 run 都是新实例，无状态残留）

Sandbox：
  ├── SandboxMiddleware.after_agent 已标记 sandbox 为 idle
  ├── sandbox_manager 后台任务每 60s 扫描
  │   → idle 超过 10min 的容器 → docker rm -f
  │   → 活跃容器不动
  └── 用户下次执行代码时重新 acquire

Memory：
  └── MemoryUpdateQueue 异步写入完成（可能在 run 结束后几秒内）
```

### 用户关闭浏览器

```
用户直接关闭标签页 / 浏览器
  → 前端没有 onFinish 回调（如果 run 还在进行中）
  → LangGraph 侧：run 会继续执行直到完成或超时
  → checkpointer 仍然会写入最终状态
  → 但 thread_runs 记录可能停留在 status=running
  → 需要后台清理任务：定期扫描超过 30min 仍为 running 的 thread_runs → 标记为 timeout
  → sandbox 不受影响（idle timeout 机制独立运行）
  → 用户下次打开同一个 thread → SDK reconnectOnMount 恢复状态
```

### 用户登出

```
用户点击登出
  → 前端调 POST /api/auth/logout
  → Gateway:
      DELETE FROM sessions WHERE token = ?
      DEL Redis session:{token}
      Clear-Cookie: session_token
  → 前端 router.push('/login')
  → 该用户的 sandbox 不立即销毁（idle timeout 自然回收）
  → 该用户的 thread/memory/skills 数据不受影响（持久化在 PG + 文件系统）
```

## 阶段 13：用户切换 / 并发隔离

### 同一时刻，用户 A 和用户 B 同时使用系统

```
用户 A 发消息                          用户 B 发消息
  │                                      │
  ▼                                      ▼
nginx auth_request                    nginx auth_request
  → Gateway 验证 A 的 cookie             → Gateway 验证 B 的 cookie
  → 注入 X-User-Id: A                   → 注入 X-User-Id: B
  │                                      │
  ▼                                      ▼
LangGraph Server (同一个进程)
  │
  ├── A 的请求:                          ├── B 的请求:
  │   runtime.context.user_id = A        │   runtime.context.user_id = B
  │   load_skills(user_id=A)             │   load_skills(user_id=B)
  │   load_memory(user_id=A)             │   load_memory(user_id=B)
  │   resolve_key(user_id=A)             │   resolve_key(user_id=B)
  │   thread_dir: users/A/threads/t1     │   thread_dir: users/B/threads/t2
  │   sandbox: A-sandbox-1               │   sandbox: B-sandbox-1
  │                                      │
  └── 完全独立的执行上下文                └── 完全独立的执行上下文
```

### 隔离保证清单

| 资源 | 隔离方式 | 隔离粒度 |
|------|---------|---------|
| thread 列表 | `threads` 表 WHERE user_id = ? | 用户级 |
| thread 消息历史 | checkpointer 按 thread_id 隔离 | thread 级 |
| thread 文件 | `users/{user_id}/threads/{thread_id}/` | thread 级 |
| memory | `user_memory` 表 WHERE user_id = ? | 用户级 |
| soul | `user_souls` 表 WHERE user_id = ? | 用户级 |
| skills 开关 | `user_mcp_configs` 表 WHERE user_id = ? | 用户级 |
| 私有 skills 文件 | `users/{user_id}/skills/custom/` | 用户级 |
| MCP 配置 | `user_mcp_configs` 表 WHERE user_id = ? | 用户级 |
| custom agents | `user_agents` 表 WHERE user_id = ? | 用户级 |
| API Keys | `user_api_keys` 表 WHERE user_id = ? | 用户级 |
| sandbox 容器 | 每用户独立容器池 | 用户级 |
| sandbox 文件挂载 | 每次 run 挂载当前 thread 目录 | thread 级 |
| 临时文件 | `users/{user_id}/threads/{thread_id}/user-data/tmp/` | thread 级 |
| session | `sessions` 表 + Redis 按 token 隔离 | 用户级 |

### 共享（不隔离）的资源

| 资源 | 原因 |
|------|------|
| 平台公共 skills | 所有用户共享，只读 |
| 平台模型配置 | 管理员统一管理 |
| 平台搜索引擎配置 | 管理员统一管理 |
| LangGraph 进程 | ASGI 并发处理，per-request 隔离 |
| PostgreSQL 实例 | 共享数据库，按 user_id 列隔离 |
| Redis 实例 | 共享缓存，按 key 前缀隔离 |
| nginx | 共享反代，按 cookie 区分用户 |

## 边界场景

### 1. 并发写同一个 thread

如果用户在两个浏览器标签页打开同一个 thread，同时发消息：
- LangGraph checkpointer 有乐观锁机制，第二个 run 会等第一个完成
- 前端 SDK 的 `useStream` 已经处理了这个（同一 thread 只允许一个活跃 stream）

### 2. thread_run 记录不完整

前端崩溃/断网时，thread_run 可能停留在 running：
- 需要后台清理任务：每 10min 扫描超过 30min 仍为 running 的记录 → 标记为 timeout

### 3. Gateway 和 LangGraph 之间的 thread 一致性

Gateway 创建 thread 写了业务表，但 LangGraph 侧创建失败：
- 先调 LangGraph 成功后再写业务表
- 或者：业务表先写 status=creating，LangGraph 成功后改为 active

### 4. 用户删除 thread 时的级联清理

删除 thread 需要清理：
- `threads` 表记录
- `thread_runs` 表记录
- LangGraph checkpoint 数据
- 文件系统 `users/{user_id}/threads/{thread_id}/` 目录
- 如果有活跃 sandbox 正在执行该 thread → 先取消 run

### 5. 平台 Key 用完 / 被封

平台统一 API Key 达到 rate limit 或被 provider 封禁：
- 所有使用平台 Key 的用户都会受影响
- Gateway 捕获 429/403 错误，返回友好提示"平台额度暂时用尽，请配置自己的 API Key"
- 有 BYOK 的用户不受影响

### 6. LangGraph 进程重启

systemd 重启 allo-langgraph：
- 正在进行的 run 会中断
- 但 checkpoint 已经持久化到 PostgreSQL，不会丢失
- 前端 SDK reconnectOnMount 会尝试重连
- 用户重新发消息即可从最新 checkpoint 恢复

### 7. Redis 重启

Redis 重启后：
- 所有 session 缓存丢失 → 下次请求会查 PG 重建缓存，用户不会被踢下线
- run:{run_id}:key 丢失 → 正在进行的 run 可能无法获取 Key → 需要重试机制
- rate_limit 计数器重置 → 短暂的限流失效，可接受
