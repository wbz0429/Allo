# Allo (元枢) 生产部署 & 多租户方案

## 决策摘要

| 决策项 | 选择 |
|--------|------|
| 部署方式 | 单机裸机部署，systemd 管理应用服务 |
| 认证 | Gateway 自管 auth，邮箱+密码，不依赖 Better Auth |
| 模型服务 | 混合模式（平台默认 + 用户 BYOK） |
| 租户隔离 | 用户级隔离优先，预留 org_id 向前兼容 |
| 数据库 | 单 PostgreSQL（Gateway + LangGraph checkpoints） |
| 缓存 | Redis（session 缓存、限流、pub/sub） |
| Sandbox | Docker 容器 per-user，资源限制 |
| Memory | 从 JSON 文件迁移到 PostgreSQL |
| 文件存储 | 服务器本地磁盘 + Docker volume |
| 搜索引擎 | 可插拔架构，平台内置 + 用户可扩展 |
| 第一版部署形态 | 1 台应用服务器，应用裸机 + sandbox Docker |
| 第一版服务器规格 | 16C / 32G Linux |
| Thread 架构 | 前端只连 Gateway，Gateway 统一转发 LangGraph |
| 私有 Skill 存储 | `users/{user_id}/skills/custom/` |
| Skill 开关状态 | PostgreSQL per-user 存储 |
| Soul | per-user 存储，用户个人画像/风格配置 |
| Custom Agents | per-user 存储 |
| 配置模型 | 平台全局默认 + 用户差异覆盖 |
| Checkpointer | PostgreSQL，作为 thread 运行状态存储 |
| Thread 与 Model | 只存 `last/default model`，运行时以当前请求参数为准 |
| Thread 与 Sandbox | thread 不强绑 sandbox，按需借用用户级 sandbox 池 |
| Sandbox 复用策略 | 每用户最多 2 个活跃 sandbox |
| 临时目录策略 | 每个 thread 独立 tmp，并注入 `TMPDIR/TMP/TEMP` |
| Thread 运行记录 | 第一版增加简化版 `thread_runs` |
| SDK 策略 | 保留 SDK 流式 + Gateway 管理 thread CRUD |
| thread_runs 记录 | 前端主动通知 Gateway |
| 流式链路鉴权 | nginx auth_request 注入 user_id |
| 不侵入 harness | harness 只接收 user_id，不承担业务编排 |

## 架构总览

```
                    ┌─────────────────────────────────────────────┐
                    │              Nginx (port 2026)               │
                    │         TLS termination + routing            │
                    └──────┬──────────┬──────────┬────────────────┘
                           │          │          │
                    ┌──────▼──┐ ┌─────▼────┐ ┌──▼──────────────┐
                    │Frontend │ │ Gateway  │ │ LangGraph Server│
                    │ :3000   │ │  :8001   │ │     :2024       │
                    └─────────┘ └──┬───┬───┘ └───────┬─────────┘
                                   │   │             │
                    ┌──────────────▼───▼─────────────▼─────────┐
                    │           PostgreSQL :5432                 │
                    │  (Better Auth + Gateway + Checkpoints)    │
                    └──────────────────────────────────────────┘
                    ┌──────────────────────────────────────────┐
                    │           Redis :6379                      │
                    │  (session cache, rate limits, pub/sub)    │
                    └──────────────────────────────────────────┘
```

核心原则：`user_id` 是所有数据的主隔离键。`org_id` 可空，默认为用户个人空间，为未来团队功能预留。

平台配置策略：

- 平台维护一份“公共底座”：默认模型、默认搜索引擎、公共 skills、公共 agent 模板
- 用户只维护自己的“差异覆盖”：自己的 skill/mcp/soul/custom agent/api key，以及公共 skill 的开关状态
- 用户不能修改平台公共底座本身，只能在允许范围内做自己的增量和开关

## 上线前提（基础设施现实约束）

当前仓库和 Docker 都跑在你的个人电脑上，这只能作为开发环境，不适合公司线上使用。第一版线上必须准备一台独立、长期在线的 Linux 服务器。

### 第一版推荐部署形态

- 1 台应用服务器
- `16C / 32G / >= 200GB SSD`
- Ubuntu 22.04 或 24.04
- 公网 IP + 域名 + HTTPS 证书
- 应用裸机部署（systemd 管理）
- 仅用户 sandbox 使用 Docker

### 这台服务器承载的服务

- `nginx`：域名接入、HTTPS、反向代理
- `frontend`：Next.js 前端
- `gateway`：FastAPI 网关、认证、租户隔离、配置 API
- `langgraph`：Agent 执行和会话流式输出
- `postgres`：用户、session、memory、配置、thread ownership、usage
- `redis`：session cache、rate limit、pub/sub
- `sandbox containers`：用户代码执行容器（唯一使用 Docker 的部分）

### 为什么不能直接用你本地电脑当线上

- 你关机、休眠、断网，服务就没了
- 本地网络和公网访问不稳定，不适合作为企业服务入口
- 数据、日志、上传文件、数据库都依赖你的个人环境，不可控
- Docker 和分支切换、本地开发操作会直接影响线上可用性
- 无法稳定做备份、监控、TLS 证书和域名接入

### 第一版容量判断

在“不自部署大模型、主要调用外部模型 API”的前提下：

- `16C / 32G` 足够支持早期 `50-200` 人试运行
- 压力主要来自：并发请求、LangGraph 会话、Postgres、Redis、sandbox 容器
- 如果 sandbox 使用频繁，优先限制每用户 sandbox 数量和空闲超时，而不是一开始做多机集群

### 后续扩展路径

- 第一步：单机裸机部署
- 第二步：数据库拆到托管 RDS / 独立机器
- 第三步：Redis 独立化
- 第四步：LangGraph / Gateway 多实例化
- 第五步：sandbox 单独迁移到独立执行节点

这个扩展路径与当前方案兼容，不会推翻第一版设计。

技术定位说明：

- `checkpointer` 保存的是 LangGraph `thread` 的运行状态快照
- 它不是用户 session，也不是长期 memory，更不是 thread 业务元数据表
- 一个 `thread_id` 会随着多轮对话不断写入/更新 checkpoints
- 它解决的是“状态恢复”和“服务重启后会话不丢”

与业务层的分工：

- `threads` 表：记录 thread 归属、标题、状态、最近活跃时间等业务元数据
- `checkpointer`：记录该 thread 的 LangGraph 内部运行状态

两者通过 `thread_id` 关联，但职责不同，不能互相替代

## Phase 2: 认证 & 用户系统

**目标**：Gateway 自管认证，用户可注册/登录，session 经 Redis 缓存验证，所有数据表带 `user_id`。

**关键决策**：不使用 Better Auth。Gateway 是认证的唯一权威源。

原因：
- Better Auth 是前端优先框架，认证权威在 Next.js 侧，与 Gateway 中心化架构冲突
- 现有 Better Auth 代码从未真正集成，只是手写的 JSON 文件 dev 实现
- 第一版需求简单（邮箱+密码），不需要完整认证框架
- Gateway 自管 auth 后，session 验证、ownership 校验、nginx auth_request 都在同一个地方

### 2.1 Gateway Auth 实现

新建 `backend/app/gateway/routers/auth.py`：
- `POST /api/auth/register` — 注册（邮箱+密码+显示名）
  - 密码用 bcrypt 或 scrypt 哈希存储
  - 注册成功后自动创建个人 Organization + membership
  - 自动创建 session，返回 session cookie
- `POST /api/auth/login` — 登录
  - 校验邮箱+密码
  - 创建 session（随机 token，写 PG + Redis）
  - 返回 httpOnly session cookie
- `POST /api/auth/logout` — 登出
  - 删除 PG session 记录
  - 删除 Redis 缓存
  - 清除 cookie
- `GET /api/auth/session` — 获取当前 session 信息
  - 返回 user_id, org_id, display_name, email
- `GET /api/auth/check` — 内部端点（供 nginx auth_request 用）
  - 验证 cookie，返回 `X-User-Id` / `X-Org-Id` 响应头
  - 无效 session 返回 401

### 2.2 数据库 Schema

新建 `backend/alembic/versions/002_auth_tables.py`：

```sql
CREATE TABLE users (
    id            VARCHAR(36) PRIMARY KEY,
    email         VARCHAR(255) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    display_name  VARCHAR(255),
    avatar_url    VARCHAR(512),
    locale        VARCHAR(10) DEFAULT 'zh-CN',
    is_active     BOOLEAN DEFAULT true,
    created_at    TIMESTAMPTZ DEFAULT now(),
    updated_at    TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE sessions (
    id         VARCHAR(36) PRIMARY KEY,
    user_id    VARCHAR(36) NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token      VARCHAR(255) NOT NULL UNIQUE,
    expires_at TIMESTAMPTZ NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_sessions_token ON sessions(token);
CREATE INDEX idx_sessions_user ON sessions(user_id);
CREATE INDEX idx_sessions_expires ON sessions(expires_at);
```

Session 策略：
- token：随机生成的不透明字符串（`secrets.token_urlsafe(32)`）
- TTL：7 天，每次活跃请求可选刷新
- 存储：PG 主存 + Redis 缓存（TTL 5min）
- 验证链路：先查 Redis → 未命中查 PG → 命中后写 Redis

### 2.3 Session 验证增强

修改 `backend/app/gateway/auth.py`：

```python
async def get_auth_context(request: Request) -> AuthContext:
    token = request.cookies.get("session_token")
    if not token:
        raise HTTPException(401, "Not authenticated")

    # 1. 查 Redis 缓存
    cached = await redis.get(f"session:{token}")
    if cached:
        return AuthContext.model_validate_json(cached)

    # 2. 查 PG
    session = await db.execute(
        select(Session).where(Session.token == token, Session.expires_at > now())
    )
    if not session:
        raise HTTPException(401, "Session expired")

    # 3. 构建 AuthContext
    user = await db.get(User, session.user_id)
    org = await get_user_org(session.user_id)
    ctx = AuthContext(user_id=user.id, org_id=org.id, role="admin")

    # 4. 写 Redis 缓存
    await redis.setex(f"session:{token}", 300, ctx.model_dump_json())
    return ctx
```

## Phase 3: 数据隔离（Harness 层）

**目标**：Harness 层的全局单例全部改为 user-scoped。Memory、MCP、Skills、Agents、Thread、文件存储按 user_id 隔离。

第一版用户化范围：

- `memory`：用户独立
- `mcp`：用户独立
- `skills`：用户独立（公共 + 私有叠加）
- `soul`：用户独立
- `custom agents`：用户独立

平台全局保留项：

- 平台默认模型配置
- 平台默认搜索引擎
- 公共 skills（仓库内 `skills/public/`）
- 平台默认 agent 模板

这是最难的阶段——需要跨越 `deerflow.*` / `app.*` 架构边界，且 harness 层不能 import app 层。

### 3.1 用户上下文传播机制

**核心问题**：LangGraph Server 当前无鉴权，不知道请求来自哪个用户。

**方案**：Nginx auth_request 代理模式

```
Frontend → Nginx → auth_request → Gateway /api/auth/check
                                      ↓ (返回 X-User-Id, X-Org-Id header)
                 → proxy_pass → LangGraph Server (带 X-User-Id header)
```

修改 `docker/nginx/nginx.conf`：
```nginx
location /api/langgraph/ {
    auth_request /internal/auth-check;
    auth_request_set $x_user_id $upstream_http_x_user_id;
    auth_request_set $x_org_id  $upstream_http_x_org_id;

    rewrite ^/api/langgraph/(.*) /$1 break;
    proxy_pass http://langgraph;
    proxy_set_header X-User-Id $x_user_id;
    proxy_set_header X-Org-Id  $x_org_id;
    # ... 保留现有 SSE/streaming 配置 ...
}

location = /internal/auth-check {
    internal;
    proxy_pass http://gateway/api/auth/check;
    proxy_pass_request_body off;
    proxy_set_header Content-Length "";
    proxy_set_header Cookie $http_cookie;
}
```

**Harness 侧**：LangGraph 通过 `RunnableConfig.metadata` 拿到 user context。

新建 `backend/packages/harness/deerflow/context.py`：
```python
@dataclass(frozen=True)
class UserContext:
    user_id: str
    org_id: str

def get_user_context(config: RunnableConfig) -> UserContext | None:
    metadata = (config or {}).get("metadata", {})
    user_id = metadata.get("x-user-id") or metadata.get("user_id")
    org_id = metadata.get("x-org-id") or metadata.get("org_id")
    if user_id:
        return UserContext(user_id=user_id, org_id=org_id or "default")
    return None
```

此模块在 `deerflow.*` 内，不 import `app.*`，只读 RunnableConfig 字典。

### 3.1.1 Gateway ↔ LangGraph 接口边界

**架构原则**：不侵入 harness。harness 是 agent 执行核心，只接收 user_id 做数据隔离，不承担业务编排。

**并发模型**：所有用户共享同一个 LangGraph Server 进程（ASGI 异步），每次 run 请求有独立执行上下文。进程级全局单例（skills/memory/mcp/config）通过 Phase 3 改为 per-request 按 user_id 动态加载。

**SDK 策略**：保留 SDK 流式 + Gateway 管理

- 流式消息（发消息 + 接收 AI 回复）：前端继续用 SDK `useStream` 直连 LangGraph，nginx `auth_request` 注入 user_id
- thread CRUD（创建/列表/删除/重命名）：前端改成调 Gateway REST API

**6 个核心操作的改造方案**：

| 操作 | 改造前 | 改造后 |
|------|--------|--------|
| 创建 thread | SDK 隐式调 LangGraph | 前端先调 Gateway 创建 → Gateway 调 LangGraph + 写业务表 → 前端再用 SDK 发消息 |
| 发消息+流式返回 | SDK 直连 LangGraph | 不变，nginx auth_request 注入 user_id |
| 列出 threads | SDK 调 LangGraph search | 前端调 Gateway → Gateway 查业务表（不调 LangGraph） |
| 删除 thread | SDK 调 LangGraph delete | 前端调 Gateway → Gateway 校验 ownership + 调 LangGraph 删 checkpoint + 删业务表 |
| 重命名 thread | SDK 调 LangGraph updateState | 前端调 Gateway → Gateway 更新业务表 + 调 LangGraph 同步 checkpoint |
| 标题自动生成 | TitleMiddleware → SSE → 前端内存 | 不变，但前端收到标题后额外调 Gateway PATCH 同步到业务表 |

**thread 创建时机**：发消息时隐式创建（和现在一致）

具体流程：
```
用户发第一条消息
  → 前端调 Gateway: POST /api/threads { thread_id, agent_name, model }
  → Gateway 校验 auth → 调 LangGraph POST /threads → 写 threads 业务表
  → 返回 thread_id
  → 前端用 SDK useStream.submit() 发消息（此时 thread 已存在于两侧）
```

**thread_runs 记录方式**：前端主动通知 Gateway

```
前端 submit() 前: POST /api/threads/{id}/runs → Gateway 写 thread_runs(status=running)
前端 onFinish 时: PATCH /api/threads/{id}/runs/{run_id} → Gateway 更新 status + finished_at
同时更新 threads.last_active_at 和 threads.last_model_name
```

**流式链路鉴权**：nginx auth_request

```
前端 SDK → POST /api/langgraph/threads/{id}/runs/stream
  → nginx auth_request → Gateway /api/auth/check → 返回 X-User-Id/X-Org-Id
  → nginx 注入 header → 转发到 LangGraph:2024
  → LangGraph 通过 runtime.context 拿到 user_id
```

**前端代码改动清单**：

## Phase 4: 模型服务（混合模式）

**目标**：平台提供默认模型服务（管理员配置 API Key，用户有 quota），用户也可配置自己的 API Key 解锁更多额度。

**Key 注入方式**：Gateway 预解析传入

流式消息发送前，前端先调 Gateway 创建 thread_run，Gateway 在此时：
1. 根据 user_id + 请求的 model_name 解析出对应 provider
2. 查 `user_api_keys` 表 → 有则解密用户 Key
3. 没有则回退到 `config.yaml` 平台 Key
4. 把解析后的 `api_key` 和 `base_url` 通过 runtime.context 传给 LangGraph

这样 harness 层的 `model_factory` 不需要查库，只需要从 context 里读 Key。不侵入 harness。

**Quota 第一版**：只记录不限制。用量数据从 thread_runs 聚合即可，不单独建 quota 表。后续再加实时拦截。

## Phase 5: Sandbox 隔离

**目标**：每个用户会话使用独立的 Docker 容器执行代码，资源受限，会话结束后销毁。

第一版语义约束：

- sandbox 是“用户级执行资源”，不是 thread 的永久宿主
- thread 不强绑定某一个固定 sandbox 容器
- 同一 thread 后续允许在同一用户的 sandbox 池中切换容器执行
- thread 的连续性主要依赖：`checkpoint + thread workspace + user context`
- 第一版默认每个用户最多 2 个活跃 sandbox

### 5.1 Sandbox Provider 架构

当前已有 `LocalSandboxProvider`（全局单例）和 `AioSandboxProvider`（Docker 容器，per-thread LRU）。生产环境需要基于 `AioSandboxProvider` 扩展为 per-user 模式。

修改 `backend/packages/harness/deerflow/sandbox/aio_provider.py`（或新建 `user_sandbox_provider.py`）：

```python
class UserSandboxProvider(SandboxProvider):
    """Per-user Docker sandbox with resource limits."""

    def __init__(self, config: SandboxConfig):
        self.max_per_user = config.get("max_sandboxes_per_user", 2)
        self.idle_timeout = config.get("idle_timeout_seconds", 600)  # 10min
        self._sandboxes: dict[str, list[Sandbox]] = {}  # user_id → [Sandbox]

    async def acquire(self, user_id: str, thread_id: str) -> Sandbox:
        # 1. 检查用户是否已有空闲 sandbox → 复用
        # 2. 用户 sandbox 数未达上限 → 创建新容器
        # 3. 已达上限 → 回收最久未用的 sandbox，创建新的
        ...

    async def release(self, sandbox_id: str) -> None:
        # 标记为 idle，启动 idle_timeout 计时器
        # 超时后销毁容器
        ...
```

### 5.3 文件挂载

用户的 workspace 目录挂载到 sandbox 容器内：

```python
volumes = {

        "mode": "rw"
    },
```

第一版挂载边界：

- `thread workspace/uploads/outputs/tmp`：当前 thread 私有，其中 `workspace/outputs/tmp` 可写，`uploads` 只读
- `users/{user_id}/skills/custom`：当前用户私有 skill，只读挂载
- `skills/public`：平台公共 skills 和脚本，只读挂载

这样 thread 运行时：

- 可以写自己的工作区和临时目录
- 可以读自己的私有能力包
- 可以读平台公共能力包
- 不能修改公共区，也不能访问其他用户目录

### 5.3.1 临时目录约束

为降低同一用户多个 thread 共用 sandbox 时的临时文件污染风险，第一版强制使用 thread 私有临时目录：

- host 路径：`users/{user_id}/threads/{thread_id}/user-data/tmp`
- container 路径：`/mnt/user-data/tmp`

执行脚本/命令时默认注入环境变量：

```bash
TMPDIR=/mnt/user-data/tmp
TMP=/mnt/user-data/tmp
TEMP=/mnt/user-data/tmp
```

并要求默认工作目录固定在：

```bash
/mnt/user-data/workspace
```

这样大部分脚本默认产生的中间文件也会被收敛到当前 thread 私有目录体系内，而不是落到共享 `/tmp`。

新建 `backend/packages/harness/deerflow/tools/search/registry.py`：

```python
class SearchEngine(ABC):
    name: str           # "tavily", "jina", "volcengine", "duckduckgo"
    display_name: str
    requires_api_key: bool

    @abstractmethod
    async def search(self, query: str, max_results: int = 10) -> list[SearchResult]: ...

class SearchEngineRegistry:
    _engines: dict[str, type[SearchEngine]] = {}

    @classmethod
    def register(cls, engine_cls: type[SearchEngine]) -> None:
        cls._engines[engine_cls.name] = engine_cls

    @classmethod
    def get(cls, name: str) -> type[SearchEngine]:
        return cls._engines[name]

    @classmethod
    def list_available(cls) -> list[str]:
        return list(cls._engines.keys())
```

```python
async def check_rate_limit(user_id: str, redis: Redis) -> bool:
    key = f"rate_limit:{user_id}"
    current = await redis.incr(key)
    if current == 1:

    return current <= MAX_REQUESTS_PER_MINUTE  # 默认 60
```

## 全链路数据流（端到端）

### 阶段 0：服务启动

服务器上有 5 个常驻进程 + 1 个按需启动的 Docker：

```
systemd 管理：
  ├── nginx          (80端口，反代所有服务)
  ├── postgresql     (5432，数据持久化)
  ├── redis          (6379，session缓存/限流)
  ├── allo-gateway   (8001，FastAPI 业务网关)
  ├── allo-frontend  (3000，Next.js 前端)
  └── allo-langgraph (2024，LangGraph agent 引擎)

Docker 按需：
  └── allo-sandbox-* (用户代码执行容器，空闲销毁)
```

启动顺序：`postgresql → redis → allo-gateway → allo-langgraph → allo-frontend → nginx`

Gateway 启动时：
1. 连接 PostgreSQL，执行 `alembic upgrade head` 确保 schema 最新
2. 连接 Redis，初始化连接池
3. 加载 `config.yaml` 平台配置（模型、搜索引擎、默认 skills）
4. 注册所有 router（auth, users, threads, memory, mcp, skills, agents, soul, api_keys, uploads, artifacts, suggestions）
5. 启动 sandbox manager 后台任务（定期扫描空闲容器）

LangGraph 启动时：
1. 读取 `langgraph.json`，注册 `lead_agent` graph
2. 调用 `make_checkpointer()` 连接 PostgreSQL（checkpoint 存储）
3. 等待请求（ASGI 异步服务）
4. 注意：LangGraph 不主动加载任何用户数据，一切都是 per-request 动态加载

### 阶段 1：用户打开浏览器

```
用户浏览器 → http://服务器IP
  → nginx 匹配 / → proxy_pass http://127.0.0.1:3000
  → Next.js 返回 HTML/JS/CSS
  → 浏览器渲染页面
```

Next.js middleware（`frontend/src/middleware.ts`）在服务端执行：
- 检查请求路径是否需要保护（`/workspace/*`、`/admin/*`）
- 检查 cookie 中是否有 `session_token`
- 有 cookie → 放行，渲染 workspace
- 无 cookie → 302 重定向到 `/login`

注意：middleware 只做 cookie 存在性检查（快速拦截），不做服务端 session 验证。真正的 session 验证在后续 API 请求时由 Gateway 完成。

### 阶段 2：认证流程

**新用户注册**：

```
浏览器 /register 页面
  → 用户填写邮箱 + 密码 + 显示名
  → 前端调 POST /api/auth/register
  → nginx → Gateway:8001
  → Gateway:
      1. 校验邮箱格式、密码强度、邮箱是否已注册
      2. bcrypt 哈希密码
      3. INSERT INTO users (id, email, password_hash, display_name)
      4. INSERT INTO organizations (id, name, slug) — 个人空间 org
      5. INSERT INTO organization_members (user_id, org_id, role='admin')
      6. 生成随机 session token (secrets.token_urlsafe(32))
      7. INSERT INTO sessions (id, user_id, token, expires_at=now+7d)
      8. SETEX Redis session:{token} → AuthContext JSON (TTL 5min)
      9. Set-Cookie: session_token={token}; HttpOnly; Path=/; SameSite=Lax
  → 返回 { user_id, email, display_name }
  → 前端 router.push('/workspace/chats/new')
```

**老用户登录**：

```
浏览器 /login 页面
  → 前端调 POST /api/auth/login { email, password }
  → Gateway:
      1. SELECT * FROM users WHERE email = ?
      2. bcrypt.verify(password, user.password_hash)
      3. 验证通过 → 创建 session（同注册步骤 6-9）
      4. 验证失败 → 401
  → Set-Cookie → 前端跳转 workspace
```

**Session 验证（后续每个 API 请求）**：

```
任意 API 请求带 Cookie: session_token=xxx
  → Gateway auth middleware:
      1. 读 cookie 中的 token
      2. 查 Redis: GET session:{token}
         → 命中：直接拿到 AuthContext(user_id, org_id, role)
         → 未命中：查 PG sessions 表
            → 有效且未过期：构建 AuthContext，写回 Redis (TTL 5min)
            → 无效或过期：401 Unauthorized
      3. 把 AuthContext 挂到 request.state.auth
      4. 后续 handler 通过 Depends(get_auth_context) 拿到当前用户
```

**nginx auth_request（LangGraph 流式链路专用）**：

```
前端 SDK → POST /api/langgraph/threads/{id}/runs/stream
  → nginx 先执行 auth_request:
      → 内部调 GET /api/auth/check (Gateway)
      → Gateway 验证 cookie → 返回 200 + X-User-Id + X-Org-Id 响应头
      → 或返回 401（nginx 直接拒绝请求）
  → 验证通过 → nginx 注入 X-User-Id/X-Org-Id header
  → proxy_pass → LangGraph:2024
```

### 阶段 3：Workspace 初始化

用户登录成功进入 `/workspace` 后，前端会并行发起多个请求加载用户数据：

```
前端 React 渲染 WorkspaceLayout
  ├── useSession()     → GET /api/auth/session     → 当前用户信息
  ├── useThreads()     → GET /api/threads           → 当前用户的 thread 列表（sidebar）
  ├── useModels()      → GET /api/models            → 可用模型列表（平台 + 用户 BYOK）
  ├── useAgents()      → GET /api/agents            → 平台默认 agents + 用户自定义 agents
  └── LocalSettings    → localStorage 读取          → 上次选择的 model/mode/agent（纯前端）
```

每个请求都经过 Gateway auth middleware，按 `user_id` 返回该用户可见的数据。

**thread 列表**来自业务表，不调 LangGraph：
```sql
SELECT id, title, status, agent_name, last_model_name, last_active_at
FROM threads
WHERE user_id = ? AND status != 'deleted'
ORDER BY last_active_at DESC
LIMIT 50
```

**模型列表**合并两个来源：
- 平台配置（`config.yaml` 中的 models）→ 所有用户可见
- 用户 BYOK（`user_api_keys` 表）→ 只有该用户可见，标记为"自有 Key"

**agents 列表**合并两个来源：
- 平台默认 agents（`config.yaml` 或 DB）
- 用户自定义 agents（`user_agents` 表 WHERE user_id = ?）

前端拿到这些数据后渲染：
- 左侧 sidebar：thread 列表（按 last_active_at 排序）
- 顶部/底部：model 选择器、mode 选择器
- 主区域：如果有 thread_id 参数则打开对应 thread，否则显示 new chat 页面

### 阶段 4：新建对话

用户点击"新建对话"或直接在输入框打字：

```
前端路由跳转到 /workspace/chats/new
  → useThreadChat() 生成一个 UUID v4 作为 threadId
  → isNewThread = true
  → useThreadStream({ threadId: undefined }) — SDK 此时不创建 thread
  → 页面渲染空白聊天区 + 输入框
  → 用户还没发消息，此时 Gateway 和 LangGraph 都没有任何操作
```

注意：thread 不是打开页面就创建的，而是用户发第一条消息时才创建。这避免了大量空 thread。

### 阶段 5：发送第一条消息

这是整个链路最复杂的一步，涉及 4 个系统协作。

```
用户在输入框输入文字，点击发送
  │
  ▼
前端 sendMessage(threadId, message, context)
  │
  ├── Step 1: 乐观渲染
  │   → 立即在聊天区显示用户消息（不等后端）
  │
  ├── Step 2: 上传文件（如有）
  │   → POST /api/threads/{threadId}/uploads (multipart/form-data)
  │   → Gateway 存到 users/{user_id}/threads/{threadId}/user-data/uploads/
  │   → 返回文件元数据 [{ name, path, size, mime_type }]
  │
  ├── Step 3: 创建 thread（仅新 thread）
  │   → POST /api/threads
  │     body: { thread_id, agent_name, default_model }
  │   → Gateway:
  │       1. 校验 auth → 拿到 user_id
  │       2. 调 LangGraph: POST /threads { thread_id } → 创建 checkpoint 容器
  │       3. INSERT INTO threads (id, user_id, title='New Chat', agent_name, default_model, ...)
  │       4. 创建 thread 文件目录:
  │          users/{user_id}/threads/{threadId}/user-data/workspace/
  │          users/{user_id}/threads/{threadId}/user-data/uploads/
  │          users/{user_id}/threads/{threadId}/user-data/outputs/
  │          users/{user_id}/threads/{threadId}/user-data/tmp/
  │       5. 返回 { thread_id, title, status }
  │
  ├── Step 4: 创建 thread_run 记录
  │   → POST /api/threads/{threadId}/runs
  │     body: { model_name, agent_name }
  │   → Gateway:
  │       1. 根据 user_id + model_name 解析 API Key:
  │          查 user_api_keys → 有则解密用户 Key
  │          没有 → 回退到 config.yaml 平台 Key
  │          都没有 → 返回错误
  │       2. INSERT INTO thread_runs (id, thread_id, user_id, model_name, agent_name, status='running')
  │       3. UPDATE threads SET last_active_at = now(), last_model_name = ?
  │       4. 返回 { run_id, resolved_api_key, resolved_base_url }
  │          (Key 通过安全通道返回，不经过前端明文，
  │           实际实现：Gateway 把 Key 写入 Redis 临时缓存 run:{run_id}:key，TTL 5min，
  │           LangGraph 通过 runtime.context.run_id 从 Redis 读取)
  │
  └── Step 5: 发送消息 + 开始流式
      → SDK useStream.submit({
          messages: [{ type: "human", content: [...] }]
        }, {
          threadId,
          streamSubgraphs: true,
          config: { recursion_limit: 1000 },
          context: {
            user_id,           // nginx auth_request 注入
            org_id,            // nginx auth_request 注入
            thread_id: threadId,
            run_id,            // 从 Step 4 拿到
            model_name,
            thinking_enabled,
            is_plan_mode,
            subagent_enabled,
            agent_name,
          }
        })
      → SDK 内部调: POST /api/langgraph/threads/{threadId}/runs/stream
      → nginx auth_request 校验 → 注入 X-User-Id/X-Org-Id
      → 转发到 LangGraph:2024
```

**Key 传递的安全设计**：

用户的 API Key 不应该经过前端。推荐链路：
1. Gateway 在 Step 4 解析出 Key 后，写入 Redis: `SETEX run:{run_id}:key 300 {api_key}`
2. LangGraph 的 `make_lead_agent(config)` 从 `runtime.context.run_id` 拿到 run_id
3. 从 Redis 读取 Key: `GET run:{run_id}:key`
4. 用这个 Key 创建模型实例
5. run 结束后 Key 自动过期

这样 Key 只在 Gateway → Redis → LangGraph 之间流转，不经过前端。

### 阶段 6：Agent 初始化与执行

LangGraph Server 收到 `POST /threads/{id}/runs/stream` 后：

```
LangGraph Server (端口 2024)
  │
  ├── 1. 解析请求
  │   → assistant_id = "lead_agent"
  │   → input = { messages: [...] }
  │   → context = { user_id, org_id, thread_id, run_id, model_name, ... }
  │
  ├── 2. 恢复 thread 状态
  │   → 从 PostgreSQL checkpointer 读取该 thread_id 的最新 checkpoint
  │   → 第一条消息时 checkpoint 为空，初始化空的 ThreadState
  │   → ThreadState 包含: messages[], title, artifacts[], todos[], sandbox 等
  │
  ├── 3. 调用 make_lead_agent(config)
  │   → 这是 graph 工厂函数，每次 run 都会调用
  │   → 从 config["configurable"] 读取:
  │       model_name, thinking_enabled, is_plan_mode,
  │       subagent_enabled, agent_name, run_id
  │
  │   → 3a. 解析模型 Key
  │       → 从 Redis 读取: GET run:{run_id}:key
  │       → 拿到 api_key + base_url
  │       → 调用 create_chat_model(name=model_name, api_key=..., base_url=...)
  │
  │   → 3b. 加载用户 Skills（per-user）
  │       → user_id 从 config["configurable"] 获取
  │       → load_skills(user_id):
  │           扫描 skills/public/ (平台公共)
  │           + 扫描 users/{user_id}/skills/custom/ (用户私有)
  │           + 从 PG 读取该用户的 skill 开关状态
  │           → 返回合并后的 enabled skills 列表
  │
  │   → 3c. 加载用户 MCP 工具（per-user）
  │       → get_cached_mcp_tools(user_id):
  │           平台默认 extensions_config.json
  │           + 用户 DB 覆盖 (user_mcp_configs 表)
  │           → 合并后加载 MCP server → 返回工具列表
  │
  │   → 3d. 加载用户 Soul（per-user）
  │       → 从 PG 读取 user_souls WHERE user_id = ?
  │       → 没有则回退到平台默认 soul 模板
  │
  │   → 3e. 加载用户 Memory（per-user）
  │       → 从 PG 读取 user_memory WHERE user_id = ?
  │       → 注入到 system prompt 中
  │
  │   → 3f. 构建 system prompt
  │       → apply_prompt_template():
  │           基础 prompt
  │           + soul 内容
  │           + memory 上下文
  │           + enabled skills 列表（name, description, location）
  │           + 工具使用说明
  │           + mode 相关指令（plan mode / thinking mode）
  │
  │   → 3g. 构建 middleware 链（15 个 middleware，按顺序）
  │       → ThreadDataMiddleware: 创建 per-user thread 目录
  │       → UploadsMiddleware: 注入上传文件路径
  │       → SandboxMiddleware: 按需获取用户 sandbox
  │       → DanglingToolCallMiddleware
  │       → ToolErrorHandlingMiddleware
  │       → SummarizationMiddleware
  │       → TodoMiddleware (plan mode)
  │       → TitleMiddleware: 首次交互后生成标题
  │       → MemoryMiddleware: 会话结束后更新 memory
  │       → ViewImageMiddleware
  │       → DeferredToolFilterMiddleware
  │       → SubagentLimitMiddleware
  │       → ResearchBudgetMiddleware
  │       → LoopDetectionMiddleware
  │       → ClarificationMiddleware
  │
  │   → 3h. 构建 StateGraph
  │       → create_agent(model, tools, middleware, system_prompt, state_schema=ThreadState)
  │       → 返回 CompiledStateGraph
  │
  ├── 4. 执行 Graph
  │   → 输入: 用户消息追加到 ThreadState.messages
  │   → 执行流:
  │
  │       before_agent middlewares (ThreadData → Uploads → Sandbox)
  │         ↓
  │       before_model middlewares (ViewImage → DeferredToolFilter)
  │         ↓
  │       model 节点 (调用 LLM API)
  │         ↓
  │       after_model middlewares (Title → SubagentLimit → ResearchBudget → LoopDetection)
  │         ↓
  │       ┌─ 模型返回 tool_calls? ─── 是 → tools 节点 (执行工具) → 回到 before_model
  │       └─ 否 → after_agent middlewares (Memory) → END
  │
  │   → 每个节点执行后，checkpointer 写入 PostgreSQL
  │   → 每个节点执行后，SSE 事件推送给前端
  │
  └── 5. 工具执行细节
      → 工具分几类:
         a. 搜索工具 (web_search, crawl)
            → 根据用户偏好选择搜索引擎
            → 调用外部搜索 API
         b. 文件工具 (read_file, write_file, ls, bash)
            → 虚拟路径 /mnt/user-data/* → 映射到 thread 目录
            → 虚拟路径 /mnt/skills/* → 映射到 public + 用户 custom skills
            → 路径解析带 user_id，防止跨用户访问
         c. 代码执行 (bash)
            → 如果是 Docker sandbox 模式 → 见阶段 10
            → 如果是 local sandbox → 直接在 host 执行（开发模式）
         d. 子任务 (task tool)
            → 创建子 agent 执行独立任务
            → 共享同一个 user context
         e. 其他工具 (present_file, ask_clarification 等)
```

### 阶段 7：流式响应回传

```
LangGraph Graph 执行中，每个节点完成后产生 SSE 事件
  │
  ├── SSE 事件类型:
  │   ├── values     — 完整 state 快照（每个节点后）
  │   ├── messages-tuple — 增量消息片段（模型 token 级流式）
  │   ├── updates    — state 差量更新（title 变化等）
  │   ├── events     — LangChain 回调事件（tool_end 等）
  │   ├── custom     — 自定义事件（task_running 子任务进度）
  │   └── tasks      — 任务状态更新
  │
  ├── 传输链路:
  │   LangGraph:2024 → SSE stream → nginx (proxy_buffering off) → 浏览器
  │   nginx 不缓冲 SSE，直接透传
  │
  └── 前端 SDK 解析:
      useStream 内部:
        ├── 解析 messages-tuple → 实时渲染 AI 回复（逐 token 显示）
        ├── 解析 values → 更新 thread.values（title, artifacts, todos）
        ├── 解析 updates → onUpdateEvent 回调（检测 title 变化）
        ├── 解析 events → onLangChainEvent 回调（tool 执行完成通知）
        ├── 解析 custom → onCustomEvent 回调（子任务进度更新）
        └── 更新 thread.isLoading / thread.status 状态
```

前端渲染链路：
```
SDK 解析 SSE
  → 更新 thread.messages（React state）
  → MessageList 组件 re-render
  → 每条 AI 消息逐 token 追加显示
  → 工具调用显示为折叠卡片（搜索结果、代码执行结果等）
  → artifacts 显示在右侧面板
  → todos 显示为任务列表（plan mode）
```

### 阶段 8：标题生成与同步

```
Agent 首次交互完成后（1条人类消息 + 1条AI回复）:
  │
  ├── TitleMiddleware.aafter_model() 触发
  │   → 用轻量 LLM 调用，根据首条对话生成简短标题
  │   → 写入 ThreadState.title
  │   → checkpointer 持久化到 PostgreSQL
  │   → SSE update 事件推送: { title: "生成的标题" }
  │
  ├── 前端 onUpdateEvent 收到 title
  │   → 乐观更新 React Query 缓存（sidebar 立即显示新标题）
  │   → 额外调 Gateway: PATCH /api/threads/{threadId} { title }
  │
  └── Gateway 收到 PATCH
      → 校验 auth + ownership
      → UPDATE threads SET title = ?, updated_at = now() WHERE id = ? AND user_id = ?
      → 业务表和 checkpoint 中的 title 保持一致
```

为什么要双写（checkpoint + 业务表）：
- checkpoint 里的 title 是 LangGraph 运行态数据，恢复会话时用
- 业务表里的 title 是列表展示用，不需要每次都去 LangGraph 查 checkpoint

### 阶段 9：后续消息（同一 thread 继续对话）

与第一条消息的区别：

```
用户在同一个 thread 继续发消息
  │
  ├── 不需要 Step 3（创建 thread）— thread 已存在
  │
  ├── Step 4 仍然需要（创建新的 thread_run）
  │   → 因为用户可能切换了 model 或 mode
  │   → 每次发消息都是一次新的 run
  │
  ├── Step 5 发消息 + 流式
  │   → SDK 调 POST /api/langgraph/threads/{id}/runs/stream
  │   → LangGraph 从 checkpointer 恢复该 thread 的最新状态
  │   → 此时 ThreadState.messages 已有之前的对话历史
  │   → 新消息追加到 messages 末尾
  │   → 重新执行 graph（model → tools → model → ...）
  │
  └── 前端 onFinish 回调
      → PATCH /api/threads/{threadId}/runs/{runId}
        { status: 'succeeded', finished_at: now }
      → 刷新 thread 列表缓存（last_active_at 变了）
      → 请求 follow-up suggestions:
        POST /api/threads/{threadId}/suggestions
        → Gateway 用轻量 LLM 生成 3 条追问建议
        → 前端在输入框上方显示
```

**model 切换场景**：

```
用户在 thread 中途切换 model（比如从 gpt-4o 切到 deepseek）
  → 前端 LocalSettings.context.model_name 更新
  → 下一次 sendMessage 时:
      Step 4 创建新 thread_run，model_name = 新模型
      Gateway 解析新模型对应的 Key
      LangGraph make_lead_agent 用新模型构建 graph
  → 对话历史不变（从 checkpoint 恢复），只是执行引擎换了
  → threads.last_model_name 更新为新模型
```

**断线重连场景**：

```
用户刷新页面 / 网络断开后恢复
  → useStream({ reconnectOnMount: true })
  → SDK 检测到该 thread 有活跃的 run
  → 调 GET /api/langgraph/threads/{id}/runs/{runId}/stream 重新加入 SSE
  → 从断点继续接收流式数据
  → 如果 run 已结束，SDK 直接加载最终 state
```

### 阶段 10：代码执行（Sandbox）

```
Agent 决定执行代码（比如用户说"用 Python 画一个正弦波"）
  │
  ├── 1. Agent 输出 tool_call: bash(command="python3 /mnt/user-data/workspace/plot.py")
  │
  ├── 2. SandboxMiddleware.before_agent 已在阶段 6 获取 sandbox
  │   → 从用户 sandbox 池中 acquire 一个容器
  │   → 如果用户已有空闲 sandbox → 复用
  │   → 如果没有且未达上限(2个) → 创建新容器:
  │       docker run -d \
  │         --name allo-sandbox-{user_id}-{short_id} \
  │         --memory 512m --cpus 0.5 \
  │         --network none \
  │         --read-only \
  │         --pids-limit 100 \
  │         --security-opt no-new-privileges \
  │         -v {thread_workspace}:/mnt/user-data/workspace:rw \
  │         -v {thread_uploads}:/mnt/user-data/uploads:ro \
  │         -v {thread_outputs}:/mnt/user-data/outputs:rw \
  │         -v {thread_tmp}:/mnt/user-data/tmp:rw \
  │         -v {user_custom_skills}:/mnt/skills/custom:ro \
  │         -v {platform_public_skills}:/mnt/skills/public:ro \
  │         -e TMPDIR=/mnt/user-data/tmp \
  │         -e TMP=/mnt/user-data/tmp \
  │         -e TEMP=/mnt/user-data/tmp \
  │         -w /mnt/user-data/workspace \
  │         allo-sandbox:latest
  │
  ├── 3. 工具执行
  │   → bash tool 收到命令
  │   → 虚拟路径验证:
  │       /mnt/user-data/* → 允许（thread 私有区）
  │       /mnt/skills/* → 允许读（公共+用户私有 skill）
  │       其他路径 → 拒绝
  │   → 路径翻译（虚拟 → host）:
  │       /mnt/user-data/workspace → users/{user_id}/threads/{thread_id}/user-data/workspace
  │       /mnt/skills/public/foo → skills/public/foo
  │       /mnt/skills/custom/bar → users/{user_id}/skills/custom/bar
  │   → 在 sandbox 容器内执行命令
  │   → 捕获 stdout/stderr
  │   → 输出中的 host 路径被掩码回虚拟路径（防止泄露服务器目录结构）
  │
  ├── 4. 结果返回
  │   → tool 返回执行结果给 agent
  │   → agent 可能继续调用更多工具（写文件、读结果、再执行等）
  │   → 生成的文件（如图片）存在 thread workspace 或 outputs 目录
  │   → 如果是 artifact（如 HTML/图片），通过 present_file tool 注册到 ThreadState.artifacts
  │
  └── 5. Sandbox 释放
      → SandboxMiddleware.after_agent 标记 sandbox 为 idle
      → 启动 idle_timeout 计时器（默认 10min）
      → 超时后容器被 sandbox_manager 后台任务销毁
      → 同一用户下次执行代码时，如果容器还在 → 直接复用
```

**Skill 脚本执行场景**：

```
Agent 读取 SKILL.md 后决定执行 skill 附带的脚本:
  → read_file /mnt/skills/public/ppt-generation/scripts/generate.py
  → bash "python3 /mnt/skills/public/ppt-generation/scripts/generate.py --output /mnt/user-data/workspace/output.pptx"
  → 脚本从 /mnt/skills/public/ 读取（只读）
  → 输出写到 /mnt/user-data/workspace/（可写）
  → 用户私有 skill 同理，路径前缀为 /mnt/skills/custom/
```

### 阶段 11：Memory 更新

```
Agent 完成一次完整交互后（after_agent 阶段）:
  │
  ├── MemoryMiddleware.after_agent() 触发
  │   → 把本次对话内容推入 MemoryUpdateQueue
  │   → Queue 有 debounce 机制（不是每条消息都立即写，而是攒一批）
  │
  ├── MemoryUpdateQueue 触发写入
  │   → 调用 MemoryStore.save_memory(user_id, data)
  │   → 第一版 MemoryStore 实现是 PostgresMemoryStore:
  │       UPSERT INTO user_memory (user_id, context_json, updated_at)
  │       INSERT INTO user_memory_facts (user_id, content, category, confidence, source)
  │
  └── 下次该用户发消息时
      → make_lead_agent 构建 prompt 时
      → 从 PG 读取 user_memory WHERE user_id = ?
      → 注入到 system prompt 的 memory 段落
      → agent 就"记住"了之前的上下文
```

Memory 的隔离保证：
- 用户 A 的 memory 只写入 user_id=A 的行
- 用户 B 的 agent 构建 prompt 时只读 user_id=B 的 memory
- 不存在跨用户 memory 泄露的可能

### 阶段 12：会话结束与清理

```
一次完整的 run 结束后，各层的收尾动作:

前端:
  ├── onFinish 回调触发
  │   → PATCH /api/threads/{threadId}/runs/{runId} { status: 'succeeded' }
  │   → invalidate React Query 缓存 ["threads", "search"]（sidebar 刷新）
  │   → 请求 follow-up suggestions
  └── 用户可以继续发消息（回到阶段 9）或关闭页面

Gateway:
  ├── 更新 thread_runs: status=succeeded, finished_at=now
  ├── 更新 threads: last_active_at=now, last_model_name=实际使用的模型
  └── Redis 中 run:{run_id}:key 自动过期（TTL 5min）

LangGraph:
  ├── checkpointer 已在执行过程中持续写入 PostgreSQL
  ├── 最终 state 已持久化（包含完整消息历史、title、artifacts、todos）
  └── graph 实例被 GC 回收（每次 run 都是新实例，无状态残留）

Sandbox:
  ├── SandboxMiddleware.after_agent 已标记 sandbox 为 idle
  ├── sandbox_manager 后台任务每 60s 扫描
  │   → idle 超过 10min 的容器 → docker rm -f
  │   → 活跃容器不动
  └── 用户下次执行代码时重新 acquire

Memory:
  └── MemoryUpdateQueue 异步写入完成（可能在 run 结束后几秒内）
```

**用户关闭浏览器**：

```
用户直接关闭标签页 / 浏览器
  → 前端没有 onFinish 回调（如果 run 还在进行中）
  → LangGraph 侧：run 会继续执行直到完成或超时
  → checkpointer 仍然会写入最终状态
  → 但 thread_runs 记录可能停留在 status=running
  → 需要后台清理任务：定期扫描超过 30min 仍为 running 的 thread_runs → 标记为 timeout
  → sandbox 不受影响（idle timeout 机制独立运行）
  → 用户下次打开同一个 thread → SDK reconnectOnMount 恢复状态
```

**用户登出**：

```
用户点击登出
  → 前端调 POST /api/auth/logout
  → Gateway:
      DELETE FROM sessions WHERE token = ?
      DEL Redis session:{token}
      Clear-Cookie: session_token
  → 前端 router.push('/login')
  → 该用户的 sandbox 不立即销毁（idle timeout 自然回收）
  → 该用户的 thread/memory/skills 数据不受影响（持久化在 PG + 文件系统）
```

### 阶段 13：用户切换 / 并发隔离

**同一时刻，用户 A 和用户 B 同时使用系统**：

```
用户 A 发消息                          用户 B 发消息
  │                                      │
  ▼                                      ▼
nginx auth_request                    nginx auth_request
  → Gateway 验证 A 的 cookie             → Gateway 验证 B 的 cookie
  → 注入 X-User-Id: A                   → 注入 X-User-Id: B
  │                                      │
  ▼                                      ▼
LangGraph Server (同一个进程)
  │
  ├── A 的请求:                          ├── B 的请求:
  │   runtime.context.user_id = A        │   runtime.context.user_id = B
  │   load_skills(user_id=A)             │   load_skills(user_id=B)
  │   load_memory(user_id=A)             │   load_memory(user_id=B)
  │   resolve_key(user_id=A)             │   resolve_key(user_id=B)
  │   thread_dir: users/A/threads/t1     │   thread_dir: users/B/threads/t2
  │   sandbox: A-sandbox-1               │   sandbox: B-sandbox-1
  │                                      │
  └── 完全独立的执行上下文                └── 完全独立的执行上下文
```

隔离保证清单：

| 资源 | 隔离方式 | 隔离粒度 |
|------|---------|---------|
| thread 列表 | `threads` 表 WHERE user_id = ? | 用户级 |
| thread 消息历史 | checkpointer 按 thread_id 隔离 | thread 级 |
| thread 文件 | `users/{user_id}/threads/{thread_id}/` | thread 级 |
| memory | `user_memory` 表 WHERE user_id = ? | 用户级 |
| soul | `user_souls` 表 WHERE user_id = ? | 用户级 |
| skills 开关 | `user_mcp_configs` 表 WHERE user_id = ? | 用户级 |
| 私有 skills 文件 | `users/{user_id}/skills/custom/` | 用户级 |
| MCP 配置 | `user_mcp_configs` 表 WHERE user_id = ? | 用户级 |
| custom agents | `user_agents` 表 WHERE user_id = ? | 用户级 |
| API Keys | `user_api_keys` 表 WHERE user_id = ? | 用户级 |
| sandbox 容器 | 每用户独立容器池 | 用户级 |
| sandbox 文件挂载 | 每次 run 挂载当前 thread 目录 | thread 级 |
| 临时文件 | `users/{user_id}/threads/{thread_id}/user-data/tmp/` | thread 级 |
| session | `sessions` 表 + Redis 按 token 隔离 | 用户级 |

**不隔离（共享）的资源**：

| 资源 | 原因 |
|------|------|
| 平台公共 skills | 所有用户共享，只读 |
| 平台模型配置 | 管理员统一管理 |
| 平台搜索引擎配置 | 管理员统一管理 |
| LangGraph 进程 | ASGI 并发处理，per-request 隔离 |
| PostgreSQL 实例 | 共享数据库，按 user_id 列隔离 |
| Redis 实例 | 共享缓存，按 key 前缀隔离 |
| nginx | 共享反代，按 cookie 区分用户 |

### 你可能没考虑到的边界场景

**1. 并发写同一个 thread**

如果用户在两个浏览器标签页打开同一个 thread，同时发消息：
- LangGraph checkpointer 有乐观锁机制，第二个 run 会等第一个完成
- 但 thread_runs 表可能出现两条 running 记录
- 建议：前端 SDK 的 `useStream` 已经处理了这个（同一 thread 只允许一个活跃 stream）

**2. thread_run 记录不完整**

前端崩溃/断网时，thread_run 可能停留在 running：
- 需要后台清理任务：每 10min 扫描超过 30min 仍为 running 的记录 → 标记为 timeout
- 在 `sandbox_manager` 的后台循环里顺便做

**3. Gateway 和 LangGraph 之间的 thread 一致性**

Gateway 创建 thread 写了业务表，但 LangGraph 侧创建失败：
- 需要事务性处理：先调 LangGraph 成功后再写业务表
- 或者：业务表先写 status=creating，LangGraph 成功后改为 active

**4. 用户删除 thread 时的级联清理**

删除 thread 需要清理：
- `threads` 表记录
- `thread_runs` 表记录
- LangGraph checkpoint 数据
- 文件系统 `users/{user_id}/threads/{thread_id}/` 目录
- 如果有活跃 sandbox 正在执行该 thread → 先取消 run

**5. 用户注销账号**

第一版可以不做，但要预留：
- 删除 users 表记录（CASCADE 删除 sessions, threads, thread_runs, memory, soul, agents, api_keys, mcp_configs）
- 删除文件系统 `users/{user_id}/` 整个目录
- 删除 LangGraph 中该用户所有 thread 的 checkpoint

**6. Skills 安装冲突**

两个用户安装同名 custom skill：
- 不冲突，因为路径是 `users/{user_id}/skills/custom/{skill_name}`
- 各自独立目录

**7. 平台 Key 用完 / 被封**

平台统一 API Key 达到 rate limit 或被 provider 封禁：
- 所有使用平台 Key 的用户都会受影响
- 建议：Gateway 捕获 429/403 错误，返回友好提示"平台额度暂时用尽，请配置自己的 API Key"
- 有 BYOK 的用户不受影响

**8. LangGraph 进程重启**

systemd 重启 allo-langgraph：
- 正在进行的 run 会中断
- 但 checkpoint 已经持久化到 PostgreSQL，不会丢失
- 前端 SDK reconnectOnMount 会尝试重连
- 如果 run 已中断，用户重新发消息即可从最新 checkpoint 恢复

**9. Redis 重启**

Redis 重启后：
- 所有 session 缓存丢失 → 下次请求会查 PG 重建缓存，用户不会被踢下线
- run:{run_id}:key 丢失 → 正在进行的 run 可能无法获取 Key → 需要重试机制
- rate_limit 计数器重置 → 短暂的限流失效，可接受
