# 多租户运行时排查与修复总结

更新时间：2026-04-01

## 一、这轮工作的目标

本轮工作的目标不是单点修 bug，而是围绕 Allo 当前多租户架构，把以下链路从“设计上应该成立”推进到“真实可运行、可验证”：

1. 本地完整启动链路
2. 认证链路
3. Frontend / Gateway / LangGraph 接口契约一致性
4. 多租户核心数据隔离
5. MCP / skills / memory / knowledge base / api keys 等配置与数据的持久化
6. marketplace 安装记录与 runtime 生效之间的真实关系


## 二、已经完成并验证的部分

### 1. 启动链路

已修复并验证：

- `make dev` 能启动 LangGraph / Gateway / Frontend / Nginx
- 本地启动脚本现在会先做 config preflight
- 启动过程增加了更清晰的日志输出

关键修复包括：

- `config.yaml` 中 `models:` 为空时，`AppConfig` 不再因 YAML `null` 直接崩溃
- `scripts/serve.sh` 增加 preflight 与开发账号种子逻辑
- `scripts/wait-for-port.sh` 增强等待期日志
- 自动识别 Homebrew nginx 路径


### 2. 认证链路

已修复并验证：

- 注册
- 登录
- session 校验
- `/workspace` 访问控制

关键修复包括：

- 前端 auth API 与后端 contract 对齐
  - `displayName -> display_name`
  - `Session` 结构与后端返回统一
- 去掉开发模式下默认 `SKIP_AUTH=1` 对真实 session 的破坏
- 增加固定本地开发账号，避免反复被历史账号状态干扰

当前开发账号：

- `dev@allo.local`
- `Password123!`


### 3. Frontend / Gateway / LangGraph 接口契约

已修复并验证：

- threads API
- auth API
- frontend thread hooks
- admin/layout session 判断逻辑

关键修复包括：

- `thread_id` / `id` 对齐
- `ThreadResponse` 改造成兼容 LangGraph SDK `Thread` 结构
- thread title 默认值与创建时序问题


### 4. 多租户主数据隔离

已做 A/B 用户真实验证并确认成立：

- `auth/session`
- `threads`
- `thread_runs`
- `knowledge-bases`
- `soul`
- `api_keys`
- `marketplace installed skills`

其中已验证的典型行为包括：

- 用户 A 创建 thread，用户 B `list` 看不到
- 用户 B 直接访问 A 的 thread 返回 `404`
- 用户 A 创建 knowledge base，用户 B 看不到
- 用户 A 写 soul，用户 B 不受影响
- 用户 A 配 API key，用户 B 不受影响


### 5. memory 租户化

这轮已修复 `/api/memory` 之前读全局 `memory.json` 的问题。

当前 `memory` API 已改为：

- 优先从 `MemoryStore`（PG-backed）按当前 `user_id` 读取
- 仅在 store 不存在时才回退旧逻辑

已做 A/B 用户验证，当前 API 不再返回同一份全局 memory 视图。


### 6. MCP 配置与执行级隔离

MCP 是本轮验证程度最高的配置型资源之一。

已完成：

- Gateway `/api/mcp/config` 切到 per-user / per-org DB 配置
- `PostgresMcpConfigStore` 能按用户返回不同 MCP 配置
- Harness MCP tools 加载路径支持按用户读取配置
- 执行级 A/B 验证已完成

执行级验证方式：

- 新增本地测试 MCP server：`backend/dev_mcp_echo_server.py`
- 为两个用户写入同名 MCP server，但 env 不同
- 实际执行 tool 后，A / B 返回不同 env 值

结论：

- MCP 当前已经不是“只有配置隔离”，而是“执行结果隔离”也成立


### 7. skills config 租户化

本轮新增：

- 表：`user_skill_configs`
- migration：`005_user_skill_configs.py`
- `PostgresSkillConfigStore`

已验证：

- 用户 A 可以禁用某个 skill
- 用户 B 不受影响

即用户级 skill toggle 已经是租户级状态。


### 8. marketplace install/uninstall 记录层

已完成并验证：

- skill install/uninstall
- tool install/uninstall
- org 级隔离

即：

- A 安装的 marketplace item，B 看不到
- uninstall 后 installed 列表会更新

## 五、当前仍未完全收口的问题

### 1. marketplace install -> runtime effect 还未彻底完成

这是目前最核心、也是剩余最难的一段。

已经确认：

- marketplace install 记录层成立
- marketplace runtime mapping 已开始补
- 真实 LangGraph runtime 也已经拿到 `user_id / org_id`

但仍未完全完成：

- `OrgInstalledSkill` / `OrgInstalledTool`
  是否真正成为 runtime 最终可用 skill/tool 集合的硬过滤条件

当前现状：

- 脚本级 skill gating 已能看出 A/B 差异
- 真实 runtime 里，installed skill 的最终效果还没完全落稳

也就是说：

- install 记录层有了
- 多租户上下文也有了
- 但 final runtime resolver 还没有完全收口


### 2. `/api/skills` 仍不是“最终用户 skill catalog”

当前 `/api/skills` 逻辑仍是：

- 全局公共 skills
- 再叠加用户 toggle

它还没有真正把 marketplace installed skills 合并进去形成统一的“当前用户 skill 表单”。

而正确模型应该是：

- built-in / assigned skills
- + marketplace installed skills
- 再结合 enabled toggle
- 形成 final runtime skills


### 3. runtime gating 需要统一 resolver

当前 install / enable / runtime 三层状态还没有完全统一进一个单一 resolver。

这是 skill / tool runtime 最后没收口的根因。


## 六、当前架构方向是否跑偏

总体没有跑偏。

这轮大部分修复都仍然遵循原来的多租户设计方向：

- Gateway 管理业务数据与隔离
- Harness 通过上下文和抽象能力决定运行时行为
- Frontend 负责传递用户态与发起请求

但也要明确：

- 有一部分“为了尽快验证真实 runtime 效果”的桥接改动是临时性质的
- 例如把 `user_id / org_id` 显式放进 LangGraph `configurable`

这类桥接不是长期最优解，但在当前阶段是合理的验证手段。


## 七、后续建议顺序

建议后续按以下顺序继续：

1. 实现统一 `skill catalog resolver`
   - built-in / install / enable 合并成单一最终视图
2. 让 `/api/skills` 与 runtime 共用同一 resolver
3. 让 `OrgInstalledTool` 也进入最终 runtime tool resolver
4. 做最终三段式验证：
   - 安装前不可用
   - 安装后可用
   - 卸载后不可用
5. 最后统一前端语义：
   - built-in: enable / disable
   - marketplace: install / uninstall
   - MCP: configure / enable / disable


## 八、当前阶段性判断

截至本轮结束，可以认为：

- 多租户主架构已经基本成立
- 核心数据隔离已大面积验证通过
- MCP 已进入执行级隔离验证通过状态
- 剩余最主要的问题，已经收敛到：
  - `marketplace install -> runtime effect` 的最终收口

也就是说，当前项目已经从“多租户主链是否成立”的阶段，进入了“配置与运行时最终语义是否完全统一”的收尾阶段。
