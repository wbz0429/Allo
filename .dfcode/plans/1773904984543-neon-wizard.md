# DeerFlow Harness — 现有功能梳理 & B2B 服务化方案

## 一、Harness 现有功能清单

### 1. Agent 编排引擎（核心）
- **LangGraph 状态图**：Lead Agent 基于 LangGraph 构建，支持 tool calling、状态流转、中断恢复
- **ThreadState**：类型化的图状态，包含 sandbox、thread_data、title、artifacts、todos、uploaded_files、viewed_images，带自定义 reducer（去重合并）
- **入口**：`deerflow.agents.make_lead_agent` → 返回 `CompiledGraph`，可直接 `.invoke()` / `.stream()`
- **关键文件**：`deerflow/agents/lead_agent/agent.py`、`thread_state.py`

### 2. 中间件管线（14 个中间件）
按顺序执行，覆盖完整的 Agent 生命周期：

| # | 中间件 | 功能 |
|---|---|---|
| 1 | ThreadDataMiddleware | 创建 per-thread 工作目录 |
| 2 | UploadsMiddleware | 注入上传文件信息到消息 |
| 3 | SandboxMiddleware | 获取/释放沙箱环境 |
| 4 | DanglingToolCallMiddleware | 修补未匹配的 tool_call |
| 5 | ToolErrorHandlingMiddleware | 捕获工具异常，转为错误消息继续运行 |
| 6 | SummarizationMiddleware | 对话超长时自动摘要 |
| 7 | TodoMiddleware | 计划模式任务追踪，摘要后重新注入提醒 |
| 8 | TitleMiddleware | 首轮对话后自动生成标题 |
| 9 | MemoryMiddleware | 异步队列更新长期记忆 |
| 10 | ViewImageMiddleware | 图片 base64 注入给视觉模型 |
| 11 | DeferredToolFilterMiddleware | 延迟加载工具的过滤 |
| 12 | SubagentLimitMiddleware | 限制并发子 Agent 数量 |
| 13 | LoopDetectionMiddleware | 检测重复 tool call 模式，3 次警告 5 次强停 |
| 14 | ClarificationMiddleware | 拦截澄清请求，中断图执行返回用户 |

### 3. 工具系统
**内置工具（11 个）**：
- `bash` / `ls` / `read_file` / `write_file` / `str_replace` — 沙箱文件操作
- `present_files` — 将文件暴露给用户（artifacts）
- `ask_clarification` — 向用户提问
- `view_image` — 读取图片供视觉模型分析
- `task` — 委派子 Agent 执行任务
- `setup_agent` — 创建自定义 Agent
- `tool_search` — 延迟工具发现（按关键词/正则搜索 MCP 工具）

**工具加载**：配置定义 → 内置工具 → 条件工具（vision/subagent/tool_search）→ MCP 工具

### 4. MCP 工具集成
- 支持 stdio / SSE / HTTP 三种传输协议
- OAuth2 令牌管理（client_credentials + refresh_token）
- 工具缓存 + 配置文件 mtime 失效检测
- 延迟加载模式：工具名可见但 schema 按需获取
- **关键文件**：`deerflow/mcp/`

### 5. 多模型管理
- 动态模型解析：通过 `use` 字段映射到任意 LangChain Chat Model 类
- 支持 OpenAI / Anthropic / DeepSeek / Google 等多家供应商
- 每模型独立能力配置：thinking mode、reasoning effort、vision
- DeepSeek 推理内容保留补丁
- **关键文件**：`deerflow/models/factory.py`

### 6. 配置系统
- YAML + JSON 双配置：`config.yaml`（主配置）+ `extensions_config.json`（MCP/Skills 状态）
- 环境变量解析（`$ENV_VAR`）
- 配置版本检查
- 单例缓存 + 热重载（`get_*_config()` / `reload_*_config()` / `reset_*_config()`）
- 覆盖范围：models、tools、sandbox、skills、memory、summarization、checkpointer、subagents、channels、tracing

### 7. 记忆系统
- LLM 驱动的长期记忆：自动提取事实、更新用户/历史摘要
- 异步去抖队列处理
- Token 感知的上下文注入（tiktoken）
- 支持全局记忆 + per-agent 记忆
- 记忆结构：user（工作/个人/当前关注）、history（近期/早期/长期）、facts（带置信度和分类）

### 8. 子 Agent 系统
- 并行任务委派：Lead Agent 通过 `task` 工具生成子 Agent
- 后台线程池执行 + 轮询完成状态
- 流式进度事件：task_started → task_running → task_completed/failed/timed_out
- 内置类型：通用子 Agent（全工具，50 轮）、Bash 专家（沙箱工具，30 轮）
- 可配置超时 + per-agent 覆盖

### 9. 沙箱系统
- **本地沙箱**：subprocess 执行，直接读写宿主文件
- **Docker 沙箱（AIO）**：HTTP API 调用容器，支持热池、副本限制、空闲超时、LRU 淘汰、跨进程锁
- 虚拟路径系统：`/mnt/user-data/*` ↔ 宿主路径映射
- 安全：路径遍历防护、命令路径白名单、输出路径掩码

### 10. Skills 系统
- `SKILL.md` 文件 + YAML frontmatter 定义技能
- 公共技能（内置）+ 自定义技能（用户安装）
- 注入 system prompt 的 `<available_skills>` XML
- 渐进式加载：Agent 按需通过 `read_file` 读取技能内容
- ZIP 归档安装（带 zip-bomb 防护）

### 11. 自定义 Agent
- 每个 Agent 独立目录：`config.yaml`（名称/描述/模型/工具组）+ `SOUL.md`（人格）+ `memory.json`
- 通过 `setup_agent` 工具或 API 创建
- 支持模型覆盖和工具组过滤

### 12. 状态持久化（Checkpointer）
- 三种后端：memory（内存）、sqlite（文件）、postgres（数据库）
- 同步 + 异步双模式
- 支持多轮对话恢复

### 13. 流式输出
- LangGraph `stream_mode="values"` 全状态快照
- 消息级更新（AI 文本、tool calls、tool results）
- 子 Agent 进度事件流
- StreamEvent 类型：values / messages-tuple / end

### 14. 嵌入式 Python Client
- 900 行完整客户端，无需 HTTP 服务即可使用
- API：stream / chat / list_models / list_skills / get_memory / upload_files / get_artifact 等
- 配置键缓存 + 懒加载 Agent

### 15. 社区工具提供者
- Web 搜索：Tavily / Firecrawl / InfoQuest（BytePlus）
- Web 抓取：Tavily / Firecrawl / Jina AI（readability 提取）
- 图片搜索：DuckDuckGo Images

### 16. 文件处理
- 上传自动转换：PDF / PPT / Excel / Word → Markdown
- Artifact 文件服务（per-thread 隔离）
- HTML → Markdown 提取（readabilipy + markdownify）

### 17. IM 渠道集成（App 层，非 Harness）
- Feishu（WebSocket 流式卡片）、Slack（Socket Mode）、Telegram（长轮询）
- MessageBus 发布/订阅架构
- 会话映射持久化
- 支持 per-channel / per-user 会话配置覆盖

## 二、B2B 服务化的现有差距

| 领域 | 现状 | 评估 |
|---|---|---|
| 认证/鉴权 | 完全没有。Gateway 所有端点裸奔，前端 Better Auth 仅骨架未接入 | 🔴 必须从零构建 |
| 多租户 | 无。全局单例配置，数据按 thread_id 隔离但无用户/组织归属 | 🔴 需要架构级改造 |
| API Key 管理 | 无。现有 API Key 仅用于出站 LLM 调用 | 🔴 需新建 |
| 速率限制 | 无。唯一的并发控制是内部 ChannelManager 信号量 | 🔴 需新建 |
| 用量追踪/计费 | 无。LangSmith tracing 仅用于调试 | 🔴 需新建 |
| Webhook | 无。IM 渠道仅入站消费 | 🟡 可选 |
| 批量处理 | 无客户端批量 API | 🟡 可选 |
| API 版本控制 | 无。所有端点 `/api/*` 无版本号 | 🟡 建议加 |
| HTTPS | Nginx 仅 HTTP | 🔴 生产必须 |
| CORS | `Access-Control-Allow-Origin: *` | 🔴 需收紧 |
| 数据存储 | JSON 文件 + 文件锁，不支持多进程并发 | 🔴 需迁移到数据库 |
| 配置隔离 | 全局单例，无 per-request/per-tenant 覆盖 | 🔴 需架构改造 |

## 三、企业办公助手 SaaS 产品方案（修订版）

### 产品定位
面向企业的 AI 办公助手 SaaS，按部门定制（文书/设计/宣发等），非技术用户友好。
先 SaaS 验证商业模式，后续提供私有化部署选项。

### 核心原则
1. Harness 核心保持统一，不为单个客户改代码，通过 Agent + Skills + 知识库实现定制
2. 前端配置引导替代 YAML，降低非技术用户门槛
3. 企业知识库（RAG）作为一等公民，支撑领域定制化
4. MCP/工具层提供生态扩展能力，对接客户内部系统

### 3.2 新增模块清单（按优先级排序）

#### 模块 A：用户认证 + 多租户（前置依赖）

**现状**：零认证，全局单例，无用户概念
**目标**：企业注册 → 员工账号 → 数据隔离

- 用户体系：Better Auth 已有骨架（`frontend/src/server/better-auth/`），需接通
  - 注册/登录（邮箱密码 + 企业邀请码）
  - 企业（Organization）→ 部门（Department）→ 用户（User）→ 角色（admin/member）
- 后端认证中间件：Gateway 全局 `Depends()`，从 session/token 解析 user_id + org_id
- 数据隔离：所有查询强制带 org_id 过滤
- 存储迁移：JSON 文件 → PostgreSQL（checkpointer 已有 postgres 支持）
- 文件存储：路径加 `orgs/{org_id}/` 前缀，后续可迁移 S3

**改动范围**：
- `app/gateway/app.py` — 加认证中间件
- `deerflow/config/paths.py` — 路径支持 org_id 前缀
- 前端 — 登录/注册页面、session 管理
- 新建 — 数据库 models（SQLAlchemy/SQLModel）

#### 模块 B：前端配置引导系统（核心体验）

**现状**：所有配置通过 `config.yaml` + `extensions_config.json`，对非技术用户不友好
**目标**：可视化配置向导，同时保留 config 文件一键导入

- 配置向导 UI（Setup Wizard）：
  - 第一步：选择模型（下拉选择，不需要知道 `use` 字段语法）
  - 第二步：选择工具（卡片式展示，开关启用，不需要编辑 JSON）
  - 第三步：选择/创建 Agent（模板库 + 自定义）
  - 第四步：上传企业知识库文档
- 配置存储：前端表单 → 后端 API → 数据库（per-org），不再依赖文件系统单例
- 高级模式：支持 YAML/JSON 导入导出（给技术用户）
- 配置 API：
  ```
  GET    /api/config                  # 获取当前租户配置
  PUT    /api/config                  # 更新配置
  POST   /api/config/import           # 导入 YAML/JSON 配置
  GET    /api/config/export           # 导出配置
  ```

**改动范围**：
- 前端 — 新建配置向导组件（`src/components/workspace/setup-wizard/`）
- 前端 — Settings Dialog 改造（当前 `settings-dialog.tsx` 已有 6 个 tab，需增强）
- 后端 — 新建 `routers/config.py`，配置持久化到数据库
- Harness — `get_app_config()` 需支持从数据库加载 per-tenant 配置（不改核心逻辑，加一层适配）

#### 模块 C：企业知识库（RAG）— 新建

**现状**：无 RAG 能力。现有 Memory 系统是对话记忆，不是知识库检索
**目标**：企业上传领域文档 → 向量化 → Agent 对话时自动检索相关知识

- 知识库管理 UI：
  - 创建知识库（按部门/主题分类）
  - 上传文档（复用现有 PDF/PPT/Excel/Word → Markdown 转换）
  - 文档列表、预览、删除、版本管理
  - 分块策略配置（chunk size、overlap）
- 后端 RAG Pipeline：
  - 文档解析 → 分块 → Embedding → 存入向量数据库
  - 向量数据库选型：Milvus / Qdrant / pgvector（推荐 pgvector，复用 PostgreSQL）
  - 检索 API：给定 query 返回 top-k 相关片段
- Agent 集成：
  - 新建 `knowledge_search` 工具（harness 层）
  - 在 system prompt 中注入检索到的知识片段（类似现有 Memory 注入方式）
  - 可配置：哪些 Agent 绑定哪些知识库
- 知识库 API：
  ```
  POST   /api/knowledge-bases                    # 创建知识库
  GET    /api/knowledge-bases                    # 列出知识库
  PUT    /api/knowledge-bases/{id}               # 更新知识库配置
  DELETE /api/knowledge-bases/{id}               # 删除知识库
  POST   /api/knowledge-bases/{id}/documents     # 上传文档
  GET    /api/knowledge-bases/{id}/documents     # 列出文档
  DELETE /api/knowledge-bases/{id}/documents/{doc_id}  # 删除文档
  POST   /api/knowledge-bases/{id}/search        # 检索（调试用）
  ```

**改动范围**：
- 后端 — 新建 `app/gateway/routers/knowledge_bases.py`
- Harness — 新建 `deerflow/knowledge/`（embedding、chunking、retrieval）
- Harness — 新建 `deerflow/tools/builtins/knowledge_search_tool.py`
- Harness — `agents/lead_agent/prompt.py` 加知识库上下文注入
- 前端 — 新建知识库管理页面

#### 模块 D：Agent 工作台（定制化核心）

**现状**：Agent CRUD 已有（`routers/agents.py`），但前端只有简单的 gallery + 聊天创建
**目标**：完整的 Agent 配置/测试/发布工作台

- Agent 模板库：
  - 平台预置模板（文书助手、设计助手、宣发助手等）
  - 每个模板 = SOUL.md + 工具组 + 绑定知识库 + 推荐 Skills
  - 用户可基于模板创建，也可从零开始
- Agent 配置 UI：
  - 人格编辑器（SOUL.md 的可视化编辑，带提示和示例）
  - 工具选择（卡片式，从已启用的工具中选择）
  - 知识库绑定（选择关联哪些知识库）
  - 模型选择（下拉，带能力说明）
  - Skills 选择
- Agent 测试：配置完直接在右侧面板测试对话
- Agent 发布：测试通过后发布给部门/全公司使用

**改动范围**：
- 前端 — 增强 `workspace/agents/` 页面（模板库、配置编辑器、测试面板）
- 后端 — `routers/agents.py` 增加模板 API、知识库绑定字段
- Harness — `config/agents_config.py` 增加 `knowledge_bases` 字段

#### 模块 E：管理后台（运营必需）

**现状**：无管理后台，所有管理通过 Settings Dialog 或直接改文件
**目标**：平台管理员 + 企业管理员双层管理

- 平台管理员（你自己）：
  - 租户管理（企业列表、配额调整、启停）
  - 用量统计（全局 token 消耗、活跃用户、热门 Agent）
  - 模型管理（全局可用模型、定价配置）
  - Agent 模板管理（预置模板的 CRUD）
- 企业管理员（客户的 IT 管理员）：
  - 成员管理（邀请/移除员工、角色分配）
  - 部门管理
  - 用量查看（本企业的消耗统计）
  - Agent 管理（发布/下架部门 Agent）
  - 知识库管理权限控制

**改动范围**：
- 前端 — 新建 `/admin` 路由（平台管理）和 `/workspace/settings/org` 路由（企业管理）
- 后端 — 新建 `routers/admin.py`（平台级）、增强 `routers/agents.py`（企业级权限）

### 3.4 计费模型

| 计费项 | 单位 | 说明 |
|---|---|---|
| 座席费 | per 用户/月 | 基础费用，按企业员工数收费 |
| LLM Token | per 1K tokens | 按模型分级（GPT-4o > DeepSeek），区分 input/output |
| 知识库存储 | per GB/月 | 文档 + 向量索引 |
| 沙箱计算 | per 分钟 | Agent 代码执行时间 |
| 文件存储 | per GB/月 | uploads + artifacts |

**套餐建议**：
- **基础版**：10 座席，500K tokens/月，5GB 知识库，适合小团队试用
- **专业版**：50 座席，5M tokens/月，50GB 知识库，含沙箱，适合中型企业
- **企业版**：不限座席，自定义配额，私有化部署选项，专属支持

### 3.5 Harness 核心改造点（保持统一，最小侵入）

Harness 是共享内核，改动必须向后兼容，不引入 app 层依赖。

| 改造点 | 文件 | 说明 |
|---|---|---|
| 配置加载适配 | `config/app_config.py` | `get_app_config()` 支持从外部注入配置 dict（不只从文件加载），per-tenant 配置由 app 层传入 |
| 路径支持租户前缀 | `config/paths.py` | `Paths` 构造函数接受 `tenant_prefix` 参数，影响 base_dir 下的目录结构 |
| 知识库检索工具 | 新建 `tools/builtins/knowledge_search_tool.py` | `knowledge_search(query, kb_ids)` → 调用向量检索，返回相关片段 |
| 知识库上下文注入 | `agents/lead_agent/prompt.py` | 类似 Memory 注入，在 system prompt 中加 `<knowledge_context>` 块 |
| Token 计量回调 | `models/factory.py` | 在 model 创建时注入 LangChain callback，记录 input/output token 数 |
| Agent 配置扩展 | `config/agents_config.py` | `AgentConfig` 增加 `knowledge_base_ids: list[str]` 字段 |

关键原则：Harness 只提供机制（mechanism），不包含策略（policy）。租户隔离、权限检查、计费逻辑全部在 app 层实现。

## 四、仓库管理策略（Fork + 分支隔离）

### 4.1 整体思路

```
bytedance/deer-flow (upstream)
        │
        │  fork
        ▼
your-org/deer-flow (origin)
        │
        ├── main          ← 纯净跟踪上游，只做 upstream sync，不写产品代码
        ├── product/dev   ← 产品开发主分支，所有 B2B 功能在这里
        ├── product/staging ← 预发布分支
        ├── product/release ← 生产发布分支
        └── feature/*     ← 功能分支，从 product/dev 切出，合回 product/dev
```

### 4.2 操作步骤

#### Step 1：GitHub 上 Fork
在 GitHub 上 fork `bytedance/deer-flow` 到你的组织（如 `your-org/deer-flow`）

#### Step 2：本地设置 remote
```bash
# 当前目录已经是 deer-flow
# 把 origin 改为你的 fork 地址
git remote set-url origin git@github.com:your-org/deer-flow.git

# 添加上游 remote
git remote add upstream git@github.com:bytedance/deer-flow.git

# 验证
git remote -v
# origin    git@github.com:your-org/deer-flow.git (fetch)
# upstream  git@github.com:bytedance/deer-flow.git (fetch)
```

#### Step 3：创建产品分支
```bash
# 确保 main 是最新的
git checkout main
git pull upstream main

# 创建产品开发主分支
git checkout -b product/dev
git push origin product/dev

# 设置 product/dev 为默认分支（在 GitHub Settings 里改）
```

#### Step 4：推送到你的 fork
```bash
git push origin main
git push origin product/dev
```

### 4.3 日常开发工作流

```
# 1. 开发新功能：从 product/dev 切分支
git checkout product/dev
git pull origin product/dev
git checkout -b feature/auth-system

# 2. 开发完成后合并回 product/dev
git checkout product/dev
git merge feature/auth-system
git push origin product/dev
git branch -d feature/auth-system

# 3. 定期同步上游更新（建议每 1-2 周一次）
git checkout main
git pull upstream main
git push origin main

# 4. 把上游更新合并到产品分支
git checkout product/dev
git merge main
# 解决冲突（主要在 app/ 和前端，harness 核心冲突应该很少）
git push origin product/dev
```

### 4.4 冲突最小化策略

为了让上游同步尽量无痛，产品代码应该遵循以下原则：

| 原则 | 做法 | 原因 |
|---|---|---|
| 不改 harness 核心 | 新功能通过新文件/新模块添加，不修改 `deerflow/` 下的现有文件 | 上游更新 harness 时不会冲突 |
| 扩展而非修改 | 用适配器/包装器模式，如 `get_app_config()` 加适配层而非改原函数 | 保持 harness API 兼容 |
| 产品代码放独立目录 | 新增模块放 `app/` 下（如 `app/gateway/routers/knowledge_bases.py`） | app 层是你的领地，上游很少改 |
| 前端新页面独立 | 新页面放新路由（如 `/admin`、`/setup`），不改现有组件内部逻辑 | 减少前端合并冲突 |
| 配置向后兼容 | 新增配置字段用 `NotRequired` / 默认值，不改现有字段语义 | 上游配置变更不影响你 |

### 4.5 哪些文件会有冲突风险

| 风险等级 | 文件/目录 | 说明 |
|---|---|---|
| 🟢 低 | `deerflow/` (harness 核心) | 你不改，上游改 → 自动合并 |
| 🟢 低 | `app/channels/` | 你不太需要改 IM 渠道 |
| 🟡 中 | `app/gateway/app.py` | 你要加认证中间件，上游可能加新 router → 手动合并 |
| 🟡 中 | `frontend/src/app/` | 你加新页面，上游也可能加 → 通常不冲突 |
| 🟡 中 | `config.example.yaml` | 你加新配置字段，上游也会加 → 手动合并 |
| 🔴 高 | `package.json` / `pyproject.toml` | 依赖变更容易冲突 → 每次同步需注意 |
| 🔴 高 | `Makefile` / `docker/` | 部署配置你可能大改 → 接受冲突 |

### 4.6 保护措施

- 在 GitHub 上设置 `product/dev` 为 protected branch，要求 PR review
- 设置 CI：`product/dev` 上跑 `make lint && make test && pnpm lint && pnpm typecheck`
- 上游同步时先在 `sync/upstream-YYYYMMDD` 分支上合并测试，通过后再合入 `product/dev`
- 给所有产品新增文件加统一注释头标记（如 `# [Product] ...`），方便区分哪些是你的代码
